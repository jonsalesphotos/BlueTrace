## 1. 背景与问题

BlueTrace 面向 BLE 传感器**数据采集**，首个落地场景是 **PPG 信号评测**。区别于普通心率手环 App，BlueTrace 的待测设备（DUT）是一台**多传感器设备**，一次会话需要同时采集多路异构信号，并且手机要能**反向控制设备行为**。

当前缺口：

- 没有现成工具能在采集 PPG 的同时，把 **ECG / IMU / 地磁 / 温度** 等多路信号同步落盘并可后期对齐。
- 评测时常需要 **DUT 与标准参考设备（如心率带）对照采集**，现有 App 不支持多设备并行 + 角色化管理。
- 研发调试需要手机能**实时控制设备**：开关某路传感器、改采样率、启停设备端算法——而不是只能被动接收。
- BLE 带宽有限，多路高频信号必须有明确的**打包/分包策略**，否则丢包严重。

## 2. 目标与非目标

### 2.1 目标（一期）

1. 在一次会话中，稳定采集 DUT 的**多路传感器数据**（PPG / ECG / IMU / 地磁 / 温度，可扩展）并各自落盘。
2. 支持 **1..N 设备并行**：单 DUT 闭环、DUT + REFERENCE 对照，代码与体验对设备数量透明。
3. 手机端可通过协议**控制 DUT**：启用/禁用传感器、设置采样率、启停设备端算法。
4. 显式呈现 **BLE 打包策略**：低频量组帧、高频量批包，并给出带宽预估。
5. 采集**全程不丢数据**：后台持续采集、异常分级提示、进程被杀后可恢复。
6. 数据可被用户**找到并导出**到公共目录。
7. 在 **Android 与 iOS** 上提供一致的产品体验。

### 2.2 非目标（本期不做）

- 服务器上传 / 远程下发指令 / 实时透传（属二期）。
- 数据的离线分析、信号质量算法评分、报表（评测分析在 App 外做）。
- 多用户账号体系、云同步、复杂权限管理。
- BLE 广播（本 App 仅作 Central）。
- 完全无人值守的开机自动重连（受平台后台限制，靠用户回 App 恢复）。

## 3. 目标用户与画像

| 画像 | 谁 | 关心什么 | 典型动作 |
| --- | --- | --- | --- |
| **算法/评测工程师**（主） | 评测 DUT 心率/信号算法的人 | 多路数据同步、对照参考设备、数据可对齐、可导出 | DUT+心率带对照采集，导出 CSV 做分析 |
| **固件/硬件工程师** | 调试 DUT 传感器与协议的人 | 实时控制设备、看原始分包流、核对帧格式 | 开关传感器、改采样率、看 Hex 包流 |
| **测试** | 验证采集稳定性的人 | 异常表达清晰、长时间不丢数据、可恢复 | 长时间采集、制造断连/关蓝牙验证恢复 |

## 4. 核心使用场景（User Stories）

> 格式：作为<角色>，我想<做什么>，以便<达成什么>。

- **US-1 多传感器单设备闭环**：作为算法工程师，我想只连 DUT 就能同时采 PPG+ECG+IMU，以便快速验证单设备多路数据。
- **US-2 对照采集**：作为算法工程师，我想同时连 DUT 和标准心率带，以便用参考心率评估 DUT 的算法准确度。
- **US-3 设备控制**：作为固件工程师，我想在采集前后随时开关某路传感器、改采样率，以便调试而不必重连设备。
- **US-4 设备端算法**：作为算法工程师，我想让 DUT 在设备端跑 HR/SpO₂/RR 等算法并回传结果，以便对比"设备端算法 vs 原始信号离线算"。
- **US-5 打包可视化**：作为固件工程师，我想看到高频批包与低频组帧的实时分包流与带宽，以便核对协议、定位丢包。
- **US-6 后台不丢数据**：作为测试，我想锁屏/切后台后采集继续，以便长时间稳定性测试。
- **US-7 异常隔离**：作为测试，我想某一路传感器或参考设备掉线时其余继续采，以便部分故障不毁掉整次会话。
- **US-8 进程恢复**：作为用户，我想 App 被系统杀掉后回到 App 能续采或重连，以便不丢失正在进行的会话。
- **US-9 找到并导出**：作为算法工程师，我想采完能看到每路数据文件并导出到电脑可访问的目录，以便后续分析。

## 5. 功能需求（按模块 · 带优先级）

> 优先级：**P0** = 一期必做，缺了不可用；**P1** = 一期应做，体验关键；**P2** = 可延后/二期。

### 5.1 权限与环境
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-PERM-1 | 检查蓝牙开关、BLE 扫描/连接权限、定位（按平台），区分硬性/建议条件 | P0 |
| F-PERM-2 | 缺硬性条件时阻断采集并给修复入口；建议条件（通知）可继续 | P0 |
| F-PERM-3 | 系统权限弹窗前先展示用途说明层 | P1 |

### 5.2 设备接入
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-DEV-1 | 按 Service UUID 过滤扫描 DUT / REFERENCE，显示名称、RSSI、MAC | P0 |
| F-DEV-2 | 角色化分配：DUT 必选、REFERENCE 可选；支持只选一个角色 | P0 |
| F-DEV-3 | 至少一个有效设备连接成功才允许开始采集 | P0 |
| F-DEV-4 | 记住上次设备，恢复时优先用已知地址定向重扫 | P1 |

### 5.3 设备配置与控制 ★核心
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-CTRL-1 | 列出 DUT 全部传感器，可逐路开关，标注高频/低频 | P0 |
| F-CTRL-2 | 每路可选采样率（来自设备能力声明） | P0 |
| F-CTRL-3 | 选择 DUT 设备端运行的算法（HR/SpO₂/RR/计步/跌倒等），结果随流回传 | P1 |
| F-CTRL-4 | 配置高频批包大小、低频组帧周期，并实时预估带宽 | P1 |
| F-CTRL-5 | 采集运行中可不停止下发控制指令，带指令日志与 ACK | P1 |

### 5.4 采集与可视化
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-COL-1 | 多传感器总览：每路一卡，含实时波形/值与在线状态 | P0 |
| F-COL-2 | 双视图切换：波形视图 ⇄ 原始分包流视图 | P1 |
| F-COL-3 | 原始分包流展示高频批包/低频组帧、包速率、丢包、CRC | P1 |
| F-COL-4 | 显示已采时长、已存行数、当前文件大小 | P0 |
| F-COL-5 | 采集中可打时间标签 | P2 |

### 5.5 会话与文件
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-FILE-1 | 每路传感器独立文件；会话级 manifest 记录起点时间、对齐锚点、设备控制配置 | P0 |
| F-FILE-2 | 采集结束展示摘要（时长、设备、各路文件与行数、异常） | P0 |
| F-FILE-3 | 历史会话列表与详情，展示传感器组合 | P1 |
| F-FILE-4 | 导出到公共目录（Android Downloads / iOS Files & 共享） | P0 |
| F-FILE-5 | 删除会话（前置确认） | P1 |

### 5.6 后台、异常与恢复
| 编号 | 需求 | 优先级 |
| --- | --- | --- |
| F-BG-1 | 后台/锁屏持续采集（平台各自机制，见架构文档） | P0 |
| F-BG-2 | 退出采集需明确确认（防误触，如长按 2 秒） | P0 |
| F-BG-3 | 异常分级提示：全局阻断（红）/ 单路降级（黄）/ 可恢复（蓝）；单路异常不影响其它路 | P0 |
| F-BG-4 | 进程被杀后回 App 可逐角色恢复或引导重扫 | P1 |

## 6. 范围与平台支持

### 6.1 一期 / 二期
- **一期**：本地多传感器采集闭环 + 设备控制 + 打包可视化 + 后台/恢复 + 导出。
- **二期**：服务器接入、周期上传、远程下发指令、实时透传、日志上报。

### 6.2 平台支持矩阵
| 能力 | Android | iOS | 说明 |
| --- | --- | --- | --- |
| 扫描/连接/多设备采集 | ✅ | ✅ | 核心一致 |
| 设备控制（开关/采样率/算法） | ✅ | ✅ | 协议一致 |
| 后台持续采集 | 前台服务 + WakeLock | Core Bluetooth 后台模式 + 状态保存/恢复 | 机制不同，体验目标一致（见 [跨平台技术说明](../architecture/BlueTrace_CrossPlatform_Notes.md)） |
| 进程被杀恢复 | 用户回 App / 启动恢复 | CBCentralManager Restoration + 用户回 App | iOS 无前台服务概念 |
| 导出 | MediaStore / 公共 Downloads | Files App / 分享 Sheet | 平台目录模型不同 |

## 7. 成功指标（建议）

| 指标 | 目标 | 怎么量 |
| --- | --- | --- |
| 采集稳定性 | 连续采集 ≥ 2 小时无中断、丢包率 < 0.5% | 长测会话统计 |
| 数据完整性 | 进程被杀后可恢复且不损坏已落盘数据 | 强杀恢复测试 |
| 多设备并行 | DUT + REFERENCE 双连接稳定运行 | 双设备 happy path |
| 跨平台一致性 | 同一流程在 Android/iOS 均可完成且体验对齐 | 双端走查 |

## 8. 约束与假设

- DUT 使用**未绑定（no bonding）**的 BLE 连接，明文 GATT；地址可能轮换，恢复需 Service UUID 兜底。
- DUT 协议为私有协议，真实帧格式对接时替换；REFERENCE 走标准 SIG Heart Rate Service（0x180D）。
- BLE 总带宽有限，多路高频信号必须靠批包/组帧控制包速率。
- iOS 后台执行受系统严格约束，"无人值守开机自连"在两端均不保证。

## 9. 术语表（Glossary）

| 术语 | 含义 |
| --- | --- |
| **DUT** | Device Under Test，待测设备，本期为多传感器自研设备 |
| **REFERENCE** | 参考设备，标准心率带等，用于对照评测 |
| **会话 Session** | 一次"开始采集 ~ 停止"，含一组设备分配与对齐锚点 |
| **角色 DeviceRole** | DUT / REFERENCE / 未来扩展，决定用哪个 Profile |
| **高频批包** | 单一高频传感器缓存 N 个采样点后批量打成一个 BLE 包 |
| **低频组帧** | 多种低频传感器按周期打成一帧同发 |
| **manifest** | 会话元数据 JSON：起点时间、monotonic 对齐锚点、设备配置、文件清单 |
| **monotonic 锚点** | `System.nanoTime()`/单调时钟基准，用于多路/多设备后期对齐 |
