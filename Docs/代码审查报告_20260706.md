## 0. 综合结论与修复波次

**总体判断**：正常路径（happy path）的工程质量是同类项目中上水平——commonMain 零平台泄漏、接口隔离、单消费协程串行化、token/i18n 纪律好、屏结构与原型一致度 ~85%。短板集中在三条线：

1. **故障恢复纵轴几乎没设防**（进程死亡/磁盘满/崩溃时机不巧）——架构 #1/#2/#3/#7 与交互 #1/#3/#5 从两个方向命中同一根因。
2. **破坏性操作与长任务的"确认-进行中-结果"三段式反馈缺失**——删除零确认、导出不可取消可穿透、勾选导出是摆设。
3. **真实 BLE 的接口形状不完整 + s7 分支合并即打断 Mock 链路**——M7 开工前必须先补。

**建议修复波次**：

| 波次 | 内容 | 对应条目 |
|---|---|---|
| ① 数据安全与崩溃（最急，改动小） | 采集循环异常兜底、manifest 原子写、幽灵运行页、僵尸服务、删除确认框 | 架构1/2/3/5/7；交互1/2/3/4/5/8 |
| ② 反馈契约 | 导出可取消+拦截、勾选导出裁决、日志 toast 接线、蓝牙页自刷新 | 交互6/7/11/12；界面1/11 |
| ③ M7 前置架构债 | BleClient 接口补形（write/特征ID/背压/decoder生命周期）、Mock 行为提升到接口、DI 可切换、会话索引入库 | 架构8/9/10/11；s7 1 |
| ④ UI 对齐与可达性 | 设计裁决跟进（红→蓝/紫→主蓝）、形状统一（图标盒/按钮胶囊）、长按结束语义、触控目标 | 界面2/3/4/7/9/10 |

## 1. 架构（15 条）

1. **[高] 采集事件循环无异常防护：一次落盘 IO 异常直接崩 App，且 `stop()` 永久挂起**
   `DefaultSessionController.kt:124`（`rs.launch { consume(ch) }`）、`:145-200`（consume/handleNotif 无 try/catch）；`AppModule.kt:53` 应用级 scope 无 CoroutineExceptionHandler。磁盘满/IO 错误 → 消费协程死亡 → 进程崩溃或 producers 挂满 Channel(64)、`stop()` 的 `ack.await()`(:283) 永不返回。改法：consume 整体 try/catch → `finishInternal(StopReason.ERROR)` 兜底；scope 加 handler 记 diagnostics。

2. **[高] manifest 覆盖写非原子 + 解析失败静默吞：会话可从数据页静默消失**
   `SessionStore.kt:30-32`（truncate-then-write）、`:42-45`（`runCatching{}.getOrNull()` 吞解析失败）。结束时刻被杀 → 截断 JSON → `list()`/`openSessions()` 都认不出 → raw 在盘上但 App 里永远不可见不报错，与"raw 是 source of truth"目标冲突。改法：临时文件 + `atomicMove`；解析失败记 WARN 并显示"损坏会话"占位。

3. **[中]⭐ START_STICKY 重启后变僵尸前台服务，虚假"正在采集"通知永不消失**（与交互 #5 同源）
   `CollectionService.kt:63-76`：onStartCommand 不校验 `status == COLLECTING`；sticky 重启后注入全新 READY controller → 永远走 `notify()` 分支；且每次 onStartCommand 都 launch 新收集器。改法：非 COLLECTING → `stopSelf(); return START_NOT_STICKY`；收集协程只起一次。

4. **[中] 会话启动路径在主线程做磁盘 IO**
   `CollectHomeScreen.kt:174-176` → `CollectViewModels.kt:102-122` → `DefaultSessionController.kt:106-110`（createDirectories×2 + writeManifest + StatFs 全在 Main）。改法：`start()` 改 suspend / 内部切 IO dispatcher。

5. **[中] `editSession` 失败回滚不完整：可能留下写坏的 manifest**
   `SessionStore.kt:94-108`：atomicMove 成功后 writeManifest 失败 → 只把目录移回，损坏 manifest 留在原位，叠加 #2 表现为"编辑一下，会话没了"。改法：临时名写好新 manifest 再逆序原子替换。

6. **[中] `SqlDelightSubjectRepository` 写操作在主线程阻塞，注入的 `io` 形同虚设**
   `SqlDelightSubjectRepository.kt:34-56`：upsert/delete/setCurrent 无 `withContext(io)`；`SubjectViewModel.kt:29-40` 从 Main 直调。改法：三个方法体包 `withContext(io)`。

7. **[中] 长按结束与"存储满自动结束"竞态：`stop()` 的 ack 可能永不完成**
   `DefaultSessionController.kt:152-155`（存储满 → cancel → return，消费循环退出）vs `:276-284`（stop 先读非 volatile status 再 send + await）。改法：循环退出前 drain channel 并 complete 未处理 Stop；或 `events.close()`。

8. **[中] `BleClient`/`SampleDecoder` 接口对真实协议有硬缺口**
   `BleClient.kt:38-53`：无写/命令通道、`BleNotification` 无特征标识、connect 失败契约未定义；decoder 是跨会话全局单例、无 `reset(deviceId)`/会话边界钩子（`AppModule.kt:77`）。改法：冻结前补 write / characteristicId / connect Result / decoder 生命周期四项。

9. **[中] 背压模型不适配真实 BLE：满则挂起生产者 + 每包一次 flush**
   `DefaultSessionController.kt:119,126` + `Writers.kt:21,50`。真实数百 notify/s × 多设备时 flush 成瓶颈 → channel 满挂起上游，而 GATT 回调线程不可挂起；`quality.droppedPackets` 恒 0（`:252`）。改法：批量 buffer + 定时/定量 flush；接口层明确溢出策略并计数。

10. **[中]⭐ 服务与共享层直接依赖 `MockBleClient` 具体类型**（与 s7 #1 呼应）
    `CollectionService.kt:40`（inject MockBleClient）+ `:48-49`（`setBluetoothOff` 是 Mock 独有 API）；`DefaultSessionController.kt:273`（commonMain 强转 Mock）。"蓝牙关→重连中"是产品行为却实现在 Mock 上。改法：把适配器状态回调提升进 `BleClient` 接口。

11. **[中] 会话列表全量扫目录 + 逐个 JSON 解析，无索引/缓存/分页**
    `SessionStore.kt:49-55` + `DataViewModel.kt:56-63`。几百会话时线性恶化；SQLDelight 在栈内却没用于会话索引。改法：落盘同时 upsert `session_index` 表，文件系统只做对账。

12. **[低] sessions 根路径三处各自拼装；DI 有无限定符的裸 `Path`/`CoroutineScope` 绑定**
    `AndroidPlatform.kt:23-26`、`MediaStoreExporter.kt:33-36`、`SettingsViewModel.kt:64-67`；`AppModule.kt:53-55`。改法：统一注入 + `named()` 限定符。

13. **[低] `exportLog` 输出流 null 仍报成功；两处 `catch(Exception)` 吞 CancellationException**
    `MediaStoreExporter.kt:109-113, 91, 114`。改法：null 流按 Error 处理并删 uri；放行取消异常。

14. **[低] 开口会话自动收尾的 endEpochMs 取下次启动时刻，中断会话时长严重虚高**
    `AppStartup.kt:49-53` + `SessionStore.kt:124`（注释自认未实现）+ `:136-154`（lineCount 恒 0）。改法：取会话夹内文件最大 mtime，兜底才用 now。

15. **[低] 测试缺口集中在异常与恢复路径**：consume IO 异常、stop 竞态、editSession 半途失败、损坏 manifest、`MediaStoreExporter` 错误路径、`AppStartup.decide`、`CollectionService` 生命周期均无测试——本报告高危项恰好全落在无测试分支上。

**架构总评**：KMP 骨架规整（零平台泄漏、接口隔离、单消费串行化），核心会话逻辑 JVM 可测。两大短板：故障路径工程化远低于正常路径；真实 BLE 接口形状未闭合。建议协议冻结前先偿还 #1/#2 与 #8/#9/#10。

## 4. s7 分支增量（12 条 + 两个专项结论）

1. **[高]⭐ 合回 main 后 Mock 采集链路事实性失效**
   `AppModule.kt:75-77`（分支上）把 `single<BleClient>` 无门控换成 AndroidBleClient，而 SampleDecoder 仍 Mock：Mock DUT 不再进扫描、真帧解不出、`CollectionService.kt:40` 仍 inject MockBleClient（蓝牙联动失灵）。改法：真实/Mock 绑定做成可切换（qualifier/调试开关），合并前先决策。

2. **[中] pullLog 中断被当成功**：4s idle 即判完成，断链同样"正常返回"→ 残缺日志存盘 + 成功 toast。改法：返回附完整性标志，断链明示。

3. **[中] 跨时区对时语义缺陷**：`syncTime()` 写手机墙钟+`timezone=0`，设备与手机时区不同时 UTC 偏 N 小时（心跳/固件日志全偏）。过零点/闰年实现正确有测试。改法：SET 带真实时区偏移或 UI 明示限制。

4. **[中] AndroidBleClient.write 忽略失败且无写完成确认**：API33+ 返回 Int 被 `(r as? Boolean) ?: true` 折成 true，不等 onCharacteristicWrite。改法：检查返回码+排队重试。

5. **[中] `lateinit var conn` 竞态可崩**：connectGatt 先注册回调、Conn 后赋值，极早回调触发 UninitializedPropertyAccessException。改法：先构造再连，或回调空安全。

6. **[中] 多包重组缓冲无上限**：`S7Frame.kt:210` 续片只校验 ID+index，不封顶 → 异常固件可致 OOM（Len 越界/死循环已防住）。改法：超 declaredLen 或 64KB 即 reset。

7. **[中] pullLog 逐字节装箱内存 ~16 倍放大 + logSink 满载 trySend 静默丢块**。改法：okio Buffer + send()/UNLIMITED。

8. **[低] 扫描时蓝牙恰好关闭会崩**：startScan 抛 IllegalStateException 只捕获了 SecurityException。改法：runCatching。

9. **[低] 调试日志遗留**：`AndroidBleClient.kt:90,106` `Log.d("BtRealBle",…)`。改法：删或 DEBUG 门控。

10. **[低] 真机 MAC 常量进 shared 生产代码**：`Device.kt:34` `S7_TEST_MAC`。改法：下沉测试 fixture。

11. **[低] per-device 资源只增不减**：conns/links/notifyFlows + 每设备常驻 collector，慢性泄漏。

12. **[低] API33+ CCC 写返回值语义混用**：与 GATT_SUCCESS 比较碰巧同值。改法：比较 BluetoothStatusCodes.SUCCESS。

**值得肯定**：CRC16-CCITT-FALSE 标准、小端正确、多包双校验+declaredLen 终检、帧层测试 4 文件 543 行；连接取消清理、"先连后入册"、poisoned 迟到应答防错配、FINE_LOCATION 处理都对。

**专项结论①合并风险**：文本冲突=零（merge-base 即 main HEAD），行为回归=中高——全部集中在 #1 的 DI 交换。main 还要 Mock 演示就必须先做可切换绑定。
**专项结论②真实 BLE 能否复用到 M7 采集链路**：不能直接复用，仅够控制台档位。可复用=BleClient 契约 + S7 帧层（质量好测试足）；AndroidBleClient 自注"v1 精简版"，缺写队列/自动重连/notify 背压（DROP_OLDEST 采集=数据缺损）/GATT 串行化框架/MTU 分包写五件事。M7 建议 Nordic Kotlin-BLE 替换传输层。
