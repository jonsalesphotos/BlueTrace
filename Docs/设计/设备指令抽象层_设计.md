# 设备抽象层设计 V2（DeviceProfile：识别一次 · 三分面 · 通道感知 BLE）

> 状态：设计稿 **V2**（2026-07-14，按深度评审修订；V1→V2 变更见 §7）。
> 动机（用户 2026-07-14）：**本项目初衷是兼容其他指令集的设备**——设备操作不应与 S7(B2A) 硬绑定。
> V1 评审结论：方向正确但只是"指令类适配层"，不足以成为可替换协议的"设备驱动层"——**BLE 通道抽象缺位、
> 双 registry 重复识别、OTA 模型带 S7 语义上移、大接口+向下转型矛盾、会话生命周期缺宿主**。V2 修正边界。

## 1. 现状盘点（V1 §1 有效，摘要）

指令/控制侧 S7 硬编码：`B2aDetect`（识别）、`S7Console`（指令会话）、`S7OtaSession`+`S7FileTrans`（OTA 传输）被 VM/编排直接构造；消息模型与 UI 按 S7 布局。**且底层 `AndroidBleClient` 同样硬编码**：扫描判定 FFE0/180D（`AndroidBleClient.kt:130`）、连接后固定 B2A FFE1/FFE2 与 HRS 2A37（`:192`）、每设备单一隐式 writeChar（`:199`）、`BleClient.write()` 无通道参数（`BleClient.kt:83`）——新协议若用不同 Service、多 Notify 或 Write With Response，必须改 AndroidBleClient，抽象层封不住协议。

好先例：解码侧 `ProtocolProfile`/`ProtocolRegistry`（02 R1–R3）稳定运行，且 `ProtocolProfile` 已有**连接后二次确认**形状（`ProtocolProfile.kt:39`）；`OtaProvisioner.readVersion` 注入证明编排可与具体指令集解耦。

## 2. 目标与非目标

**目标（验收口径，比 V1 更严）**

- 新增一种协议设备 = **只新增一个协议实现包 + 一行注册**，不修改框架代码（BLE 客户端、编排、UI 任何一行）。
- 设备身份**只解析一次**（统一 `DeviceProfileCatalog`），数据面/控制面/固件升级面共享同一识别结果。
- BLE 层通道感知、协议无关：服务/特征/写类型由 profile 的 `GattSpec` 声明，多 Notify 分流、写按特征寻址。
- 能力以**分面是否存在（null 与否）**表达，无"能力集合与方法不一致"的窗口，无运行时 Unsupported，大幅减少向下转型。
- 长事务（OTA/会话）由**应用级宿主**持有，不落 viewModelScope（呼应架构优化文档"OTA/长事务应由应用级宿主或前台服务持有"）。

**非目标**

- 不重写 S7 编解码（`S7Frame`/`S7Messages`/`S7FileTrans`/`S7Console` 被适配器包装）；
- 不动解码侧 `shared.protocol.registry` 的内部实现（其 registry 改由 Catalog **派生**，识别真源唯一化，接口不变）；
- 不为假想设备过度设计——接口形状以 S7 + 一个**异构** Mock 第二协议（W6 验收）能落地为准。

### 3.1 统一识别：DeviceProfile + DeviceProfileCatalog（shared/commonMain 新包 `io.bluetrace.shared.device`）

```kotlin
/** 协议通道声明：BLE 层照此发现/订阅/写，协议知识不进 BleConnection 实现。 */
data class GattSpec(
    val serviceUuid16: String,                 // 主服务（16bit 短码，口径同 B2aDetect.extract16）
    val notifyChar16s: List<String>,           // 需订阅的 Notify 特征（可多个）
    val writeChar16: String?,                  // 默认写特征（无下行的纯广播设备为 null）
    val writeWithResponse: Boolean = false,
)

/** 广播识别声明（先按广播 service 表匹配；复杂匹配交 matches 覆写）。 */
data class ScanSpec(val advertisedService16s: List<String>)

interface DeviceProfile {
    val profileId: String                      // 与解码侧同一常量（PROFILE_S7…）
    val scanSpec: ScanSpec
    val gattSpec: GattSpec
    fun matches(device: ScannedDevice): Boolean = // 默认=广播 service 命中，S7 可覆写复用 B2aDetect
        device.advertisedServices.any { it in scanSpec.advertisedService16s }
    /** 连接后二次确认（服务表校验；同解码侧 ProtocolProfile 先例）。false=识别撤销。 */
    fun confirm(discoveredService16s: List<String>): Boolean =
        discoveredService16s.contains(gattSpec.serviceUuid16)
    val dataPlane: DataPlaneFactory?           // 采集解码面：现 ProtocolProfile 挂入
    val controlPlane: ControlPlaneFactory?     // 控制面：§3.2
    val firmwareUpdate: FirmwareUpdateFactory? // 固件升级面：§3.3
}

class DeviceProfileCatalog(private val profiles: List<DeviceProfile>) {
    fun identify(device: ScannedDevice): DeviceProfile? = profiles.firstOrNull { it.matches(device) }
    fun byId(profileId: String): DeviceProfile? = profiles.firstOrNull { it.profileId == profileId }
    /** 解码侧 ProtocolRegistry 由此派生（识别真源唯一），迁移期解码代码零改动。 */
    fun toProtocolRegistry(): ProtocolRegistry = ...
}
```

### 3.2 控制面：能力分面（废除 capabilities 集合 + 大接口）

```kotlin
/** 分面是否为 null 即能力是否存在——无 Unsupported 运行时异常，无"集合与方法不一致"窗口。 */
interface DeviceControl {
    val info: DeviceInfoOps?          // suspend fun get(): DeviceCmdInfo
    val battery: BatteryOps?          // suspend fun percent(): Int
    val timeSync: TimeSyncOps?        // suspend fun sync()
    val logs: LogOps?                 // suspend fun pull(onProgress): ByteArray
    val power: PowerOps?              // suspend fun reboot()/powerOff(): Boolean（"已发送+尽力确认"语义）
    val vendor: VendorOps?            // 厂商扩展面（marker）：S7 = 自定义对时/用户信息/找表
    fun close()
}
```

- **UI 通用区**只消费六个通用分面的存在性与语义操作；**厂商专属 UI 块**（S7 的自定义对时/用户信息编辑）消费 `vendor`——`vendor as? S7VendorOps` 的类型判定只发生在"本来就是 S7 专属"的 UI 块内，这是受限的、语义正确的向下转型（与 V1"通用操作也要转型"不同）。
- `DeviceCmdInfo(swVer/hwVer/model, extras)`、`DeviceCommandException(Timeout / DeviceError(code, codeName))` 同 V1；**codeName 在协议实现构造异常时解析**（通用层零 S7 码表依赖）。

### 3.3 固件升级面：FirmwareUpdateStrategy（整体策略，不是文件传输）

```kotlin
/** 通用层只看得到：粗粒度阶段 + 进度 + 终态。包解析/校验/传输/Bootloader 切换/重连/验证全在协议实现内。 */
enum class FwUpdatePhase { Preparing, Transferring, Applying, Verifying, Done }
data class FwUpdateProgress(val phase: FwUpdatePhase, val percent: Int, val bytesPerSec: Long, val detail: String?)
sealed interface FwUpdateResult { data class Success(val versionAfter: String?) : FwUpdateResult
                                  data class Failed(val summary: String, val percentAtFailure: Int) : FwUpdateResult }

interface FirmwareUpdateStrategy {
    /** pkg 为协议自定包类型的载体（S7=zip 解析出的文件清单；他协议自定），由各协议的 loader 产出。 */
    suspend fun run(pkg: FwPackage, onProgress: (FwUpdateProgress) -> Unit): FwUpdateResult
    /** 手动中止的协议正确善后（S7=停发切片+断链等 61s 看门狗；他协议=各自语义）。 */
    suspend fun abort()
}
```

- **S7 实现吸收现有 `OtaProvisioner` 全部编排**（下载→等自复位→扫描回连→读版本；固件门控停止语义、"永不补发 STOP"红线原样内迁）——"同一设备 ID 自复位回连"从此是 **S7 策略的私有假设**，Bootloader 换地址/换服务/需激活命令/双分区确认的设备各写各的策略。
- `MultiOtaController` 瘦身为**队列壳**：排队/电量门槛/重试/取消/日志，逐台调 `strategy.run`；串行编排语义与现有测试保留。
- 现有细粒度 `OtaPhase`（扫描回连等）成为 S7 策略内部状态，经 `FwUpdateProgress.detail` 透传给 UI 日志（文案不丢）。

### 3.4 BLE 通道抽象：BleConnection（W1 落地，= 架构优化线 P0 传输地基合流）

> **实现选型（2026-07-14 用户拍板，D2 恢复）**：Android 主用 **Nordic Kotlin-BLE-Library 2.0**、iOS 用 **IOS-BLE-Library**——两库是平台库，跨平台统一层就是本节 `BleClient`+`GattSpec` 契约。切换走"并存双实现 + 真机 A/B 闸门"（W1.5/W1.6），自写 `AndroidBleClient` 过闸前保留为回退；评估与必验清单见 [`BLE库切换_Nordic评估.md`](BLE库切换_Nordic评估.md)。

- `connect(device, spec: GattSpec?)`：按 spec 发现服务、订阅**多个** Notify、登记**多写特征**；`spec=null` 走现状 B2A→HRS 探测（迁移期行为等价）。
- `write(deviceId, bytes, char16: String? = null, withResponse: Boolean? = null)`：默认参数 = 现状默认写特征/无响应写（现有调用零改动）；逐帧写流控、BUSY 退避、取消清理等**真机验证过的坑一律不动**。
- `BleNotification.characteristicId` 字段已预留——Android 实现补填，多特征协议按此分流。
- 扫描层去识别化：`AndroidBleClient.toScanned` 不再判 PROFILE_S7/HRS（识别归 Catalog，W2 迁移）。

| **W4** | `FirmwareUpdateStrategy` + S7 策略（吸收 OtaProvisioner；停止语义/STOP 红线内迁）+ MultiOtaController 队列壳化 | 中——OTA 行为语义搬迁，测试全量护航 |
| **W5** | UI 按分面显隐；S7 专属块走 vendor；B2aDetect 引用清零 | 中——UI 回归量大，真机过一遍 |
| **W6** | 异构 Mock 第二协议（**不同 Service/特征 + 缺一项能力 + 不同 OTA 重连行为**）双栈并存 | 低——纯验收；判据=只新增协议包+一行注册 |

## 5. 边界与风险

1. **W1 是最高风险波**：`AndroidBleClient` 的写流控（逐帧串行+onCharacteristicWrite 节流）、连接取消清理（泄漏连接占死设备）、CCC 订阅时序都是真机踩坑修出来的——W1 只做**加法**（spec 路径），默认路径字节级等价，真机回归后才进 W2。
2. **电源命令判据**（不变）：S7 电源类不回包、传输中被固件门控吞（`OTA/OTA中止_固件行为审查.md`）——该语义整体内迁 S7 FirmwareUpdateStrategy/PowerOps，通用层不感知。
3. **识别撤销路径**：confirm 失败（广播匹配但服务表不符）→ 会话不建立、UI 报"设备识别失败"——W2 定义统一错误流。
4. **两侧 registry 迁移期**：解码侧接口不动、实现由 Catalog 派生；若出现顺序敏感差异（同设备命中不同 profile），Catalog 单点排序即真源。
