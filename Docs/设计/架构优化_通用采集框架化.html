BlueTrace 通用采集框架化——架构分析与实施计划

2026-07-07 架构分析会话产出。勘察底稿 = 10 智能体工作流（7 子系统精读 + 平台/协议演进/数据 3 视角批判）。 决策与执行期笔记的活文档 = context/架构优化_实施笔记.md；本文是定稿快照（盲区 + 决策 + 计划 + explainer/handoff/quiz）。 真源口径不变：/SPEC.md ＞ 原型；协议参考 = S7B2A/S7协议共识规格.md。

0. TL;DR

  • 目标：BlueTrace 从"S7 采集 App"演进为通用 BLE 采集框架——换项目 = 换协议包，其余零改动。
  • 现状判定：注册式架构（02 R1–R3）方向正确但对主协议线空转——B2A 没有 Profile，识别/订阅/门控三面硬编码旁路；真正的协议锁不在解析器，而在 DecodedStream 闭合枚举（数据词汇表）与控制台整条线。
  • 终选方案 = B-lite（D7）：ProtocolProfile 拆「数据核（ScanSpec/通道表/流表/能力集）+ 行为工厂（解析器/DeviceOps）」，扫描、订阅、UI 门控、存储列全部消费数据核；不做序列化/热下发层（D1 已定发版制）。
  • 8 项决策全部拍板（D1–D8，见 §4），执行分 6 个波次（P0 传输地基 → P5 M7 联动，见 §6.6）。
  • 一条边界必须记住：D6——包内时间语义（样本时间戳/seq/间隔）归采集协议定义，App 不介入。传输层只给包打接收时间标签 + raw hex 原样落盘；协议是否带设备时间戳决定跨设备对齐/丢包核算能力，其后果随协议走，不是 App 技术债。
  • 可信度背书：本文的代码事实断言经 codex 独立只读交叉核对（13 条承重断言 11 CONFIRMED / 2 PARTIAL / 0 REFUTED，无一推翻），详见文末 附录 A。

1. 目标与范围

用户提出的 8 条目标（评估一切设计的基准）：

  1. 通用采集框架：换项目 = 换协议解析/处理库，改动文件越少越好（当前锁定 B2A）。
  2. 中间件负责注册协议：UUID、设备名、扫描广播特征等。
  3. DUT 配置抽离；UI 与解析分离；不支持的功能 Compose 不显示（能力门控）。
  4. 采集/OTA 等 BLE 交互放前台服务，UI 只显示（已确认 = MVVM，缺口在逻辑宿主）。
  5. 扫描模式确认（系统节流/mute——已确认，见盲区 B1）。
  6. 数据解码/打包；存储/导出位置；按时间+MAC 命名；采集人/场景事后可改。
  7. 后期 WLAN：推数据上服务器、获取解析器（验收档位已定 = 发版制，D1）。
  8. 防后台杀 + BLE 长连接完整清单（验收已定 = 息屏长采+自动回连，D5）。

题外目标（对架构有约束力）：同模式做嵌入式 BLE 智能网关——数据核设计是双栈复用的前提（见盲区 A5）。

3. 盲区清单（unknown unknowns）

用户原始清单之外、会实质改变方案的发现。编号规则：D=数据级（错了数据作废）、A=架构级（错了推倒重来）、X=交互/合规级。

数据级（🔴 已采数据事后不可修复）

  • D1 多 DUT 数据无归属：3 台 DUT 混写同一 raw/dut.hexlog（按 DeviceKind 分文件），行内无设备标识、CSV 无设备列、DecodedSample 无 deviceId；B2A 有状态多包重组在混流下会跨设备错拼——多 DUT 会话等于白采。→ 决策 D2 修复。
  • D2 时间轴无权威：全链路只有手机墙钟，HRS 的 deviceTsUs 是接收时刻冒充，批包内 N 样本共享同一时戳。→ D6 定分层：包内时间归协议，App 只打接收标签、不反推不校准；deviceTsUs 有无与精度随协议解析结果走。
  • D3 MAC 身份无 RPA 后路：设备 id/命名/登记/重连全锚 MAC，无 bonding；固件开 privacy 即静默断裂。→ 决策 D3（固件承诺 static）。
  • D4 droppedPackets 计数点错位：真实丢包口在 MutableSharedFlow(256, DROP_OLDEST)——tryEmit 恒成功、溢出静默；manifest 计数的 Channel 层几乎不丢，是假安全感。→ P2 补计数。
  • D5 会话主键 = 文件夹名：事后编辑 = 重命名 = 改主键；导出副本/未来服务器去重无稳定键。→ P2 加 sessionUuid。
  • D6 hexlog 可重放承诺断裂：行格式无特征标识且 characteristicId 未透传，多特征协议（B2A 扩展大概率）无法离线重放路由。→ P0 透传 + P2 格式 v2。
  • D7 无 schema 版本锚：profileId 裸字符串、manifest 不记解析器版本——M7 扩展后旧数据被新解析器静默错解。→ P2 版本化（M7 前最后窗口）。

架构级（🟡 换协议时推倒重来）

  • A1 注册表空转（见图 1）：换协议动 ≥5 处。→ P1 修复。
  • A2 数据词汇表闭合：DecodedStream（7 值含 HR）与 CollectType（6 个采集开关，不含 HR）是两个写死在 domain 的固定枚举——解析器插拔了，"事件能说什么词"没插拔；新传感器流 = 改 domain（枚举）+存储（SessionRecorder 以 DecodedStream 为 CSV writer key）+UI（CollectTypeSheet 枚举直渲）+manifest（sampling 表达）四层。这是比 UUID 更深的协议锁。→ D7/P1 开放为流表。
  • A3 控制台归属未拍板：S7Console 无接口、Compose 直传协议常量，OTA 若现在做会长在 viewModelScope 和 S7 专属类型上。→ 决策 D4（DeviceOps 语义层）。
  • A4 "服务器下发解析器"与静态注册正面矛盾：02 设计明文非目标，KMP 堵死动态代码路线。→ 决策 D1 定档"发版制"，矛盾解除。
  • A5 网关复用的是设计不是代码：Linux SBC 可跑 KMP 层；MCU/C 生态只有"协议知识做成数据"才能双栈共享。→ B-lite 数据核即伏笔。

交互/合规级（🟠）

  • X1 扫描三连环：8.1+ 息屏时无过滤扫描零结果；连续 30 分钟降级 opportunistic；5 次/30s 节流多为静默（且 [AndroidBleClient.kt:96] 注释把错误码搞错：5=OUT_OF_HARDWARE_RESOURCES，6 才是 SCANNING_TOO_FREQUENTLY）。startScan(null, LOW_LATENCY) 是最坏组合。→ P0/P1 ScanFilter 化。
  • X2 定位开关 OFF = 扫描空列表：API 31+ 要求权限已授且系统定位开关打开；现状只查权限、LOCATION 标 SUGGESTED，全仓无 isLocationEnabled。→ P3 升 HARD+补检查。
  • X3 长连接四件套空白：connectionPriority / 断链发现延迟（按样本流静默判卡死）/ autoConnect 托管回连 / FGS 类型合规（A15 起 dataSync 每日 6h 配额——M9 上传不能走 dataSync 长跑）。→ P0/P3/M9 预警。
  • X4 ROM 保活矩阵：电池优化豁免只是四件之一（MIUI 自启动、上划直杀+onTaskRemoved 未处理、华为启动管理）；CDM 是正规通道但按台关联 UX 成本高。→ P3 专项 + CDM 入 backlog。
  • X5 数据生存与合规：卸载即全灭无提示；导出 zip 明文受试者信息落公共目录。→ 决策 D8 维持明文（内部工具接受，M9 前重议）。

4. 决策记录（2026-07-07 采访，8 问 8 答）

# 决策 拍板 关键推论
D1 M9"服务器获取解析器"验收档位 发版接新协议，服务器只做分发/版本管理 方向 C（DSL 解释器）出局；数据核仍值得做但不要求可序列化
D2 多 DUT 同采 真需求，按设备分文件 hexlog/CSV per-device、DecodedSample 加 deviceId、manifest 按设备挂文件——并入格式 v2 一次破格
D3 设备身份锚 固件承诺 static address，App 维持 MAC 「不开 RPA」写入固件侧冻结项；第三方参考设备残余风险接受
D4 控制面归属 独立 DeviceOps 语义层 UI 只见 syncTime/pullLog 等语义动作；S7Console 改造为实现；OTA 宿主下沉应用级/FGS
D5 长连接验收 息屏长采 + 断链自动回连 席位状态机、会话分段+gap、FGS 补 location、WakeLock 评估、ROM 专项必做
D6 时间职责分层 包内时间语义归协议，App 只打接收标签 App 不反推间隔/不校准设备钟/不排序包内样本；协议带不带时间戳决定跨设备对齐能力，后果随协议走；App 本分=接收标签+raw hex+自身缓冲丢包计数
D7 方案方向 B-lite：数据核先行 数据核 = 普通 data class；不做 @Serializable/registry_hash（服务器线立项再升级）
D8 受试者合规 维持明文，内部工具接受 风险自担在案；M9 开工前必须重议（最低传输加密+代号化评估）

遗留三项：#9 Nordic 替换时机（本计划推荐续用自写 GATT，与归档评估 D2 口径相左待确认）、#10 S7 功能掩码逐位含义（找固件侧要）、#11 CDM 可选增强（backlog）。

5. 三方案与终选

A 静态注册表补全 B-lite 数据核先行（终选） C 描述式 DSL+解释器
Profile 形态 行为接口补字段 数据核 + 行为工厂两层 纯声明式描述文件
数据词汇表 闭合枚举 + CUSTOM 逃生舱 开放流表 StreamDescriptor 开放（描述文件内）
换协议改动 1 包 + DI 一行（但新流改四层） 1 包 + DI 一行（含新流） 服务器下发不发版
抗节流 ScanFilter 需另加专用机制 ScanSpec 天然可转 同 B
服务器下发 不可 配置级可（升级后） 协议级可
网关双栈复用 设计复用 数据核可为共同真源（升级后） 描述文件即真源
成本 最低 中 高一个数量级
淘汰原因 "换协议只换库"差最后一公里 —— D1 定发版制，无硬需求

6. 实现计划

6.1 数据模型（Data Model）

协议数据核（shared.protocol.registry，普通 data class）：

// id 带版本段: "SKG.S7.B2A@1" —— manifest 记录, 离线重放按版本取解析器(盲区 D7)
data class ProtocolId(val vendor: String, val proto: String, val major: Int)

// 声明式扫描特征: 可转 Android ScanFilter(硬件过滤 = 抗节流 + 息屏可扫, 盲区 X1)
data class ScanSpec(
    val serviceUuids: List<String>,          // 广播 16/128 位服务
    val namePrefixes: List<String> = emptyList(),
    val manufacturerId: Int? = null,
)

// 数据词汇表开放(盲区 A2): 替代 DecodedStream/CollectType 闭合枚举
data class StreamDescriptor(
    val streamId: String,                    // "ppg_g" / "acc" / "hr" / 新协议自声明
    val labelToken: String,                  // UI 文案 token(恒英文, i18n 由 res 层解析)
    val csvName: String,                     // 落盘文件名段
    val columns: List<String>,               // CSV 列(不含框架统一的接收时间列 epoch_ms)
    val defaultEnabled: Boolean,
    val uiOrder: Int,
    val colorSlot: Int,                      // 主题色槽位序号, 非硬编码色值
)

interface ProtocolProfile {                  // v2 = 数据核 + 解析工厂
    val id: ProtocolId
    val scanSpec: ScanSpec                   // matches() 保留为默认实现(按 scanSpec 判)
    val notifyChannels: List<ChannelId>
    val writeChannel: ChannelId?
    val streams: List<StreamDescriptor>      // 新增
    val capabilities: Set<Capability>        // 新增: 框架侧词表(COLLECT/TIME_SYNC/LOG_PULL/OTA/PERSON_RW/...)
    fun confirmByServices(serviceUuids: List<String>): Boolean
    fun createParser(channel: ChannelId): ChannelParser
}

控制面（shared.protocol.ops，D4）：

interface DeviceOps {                        // UI 只见语义动作, 按 supported 渲染
    val supported: Set<Capability>
    suspend fun syncTime(spec: TimeSyncSpec): OpResult
    suspend fun pullDeviceLog(sink: LogSink): OpResult
    suspend fun reboot(): OpResult
    // OTA 后续扩展; 宿主 = 应用级 scope/前台服务, 不挂 viewModelScope
}
// DeviceOpsRegistry: ProtocolId -> (BleClient, deviceId) -> DeviceOps

6.2 接口变更

接口 变更 消费方

6.3 状态机

OTA（后续，D4 推论）：宿主 = 应用级 scope + 前台服务；IDLE → TRANSFER(priority HIGH + 写队列流控) → VERIFY → APPLY；依赖 P0 的写结果消费——现状 WWR 不消费返回值，OTA 连发必丢。

6.4 用户可见行为变化

  1. 扫描页：按注册协议做系统级过滤扫描（保留"显示全部"开关兜底 nRF 模式）；息屏可继续扫；被系统限流时显示提示条（透传 onScanFailed）。
  2. 采集运行页：断链设备显示"回连中"徽标而非消失；采集类型抽屉由 Profile 流表数据驱动——协议不支持的流不再出现。
  3. 控制台：操作按 DeviceOps.supported 渲染——换协议自动隐藏不支持项；对时/拉日志行为不变。
  4. 会话详情：显示分段与 gap；质量三计数（流溢出/通道溢出/malformed）。
  5. 权限/环境：定位升 HARD 并检查系统定位开关；新增 ROM 保活引导页（检测厂商，给自启动/上划加锁/电池优化三步）。
  6. 数据：文件夹命名 5 段规则不变；事后编辑采集人/场景交互不变（主键改 sessionUuid，用户无感）。

6.5 风险点

# 风险 处置
R1 D6 分层：包内时间归协议——协议若不带时间戳/seq，跨设备对齐/间隔/丢包核算能力受限 非 App 技术债：App 只打接收标签+raw hex 原样落盘；能力后果随协议设计走，M7 定协议时权衡
R2 文件格式 v2 直接替换不迁移（demo 口径，存量真实数据少） 破格一次到位（sessionUuid/per-device/版本锚同批）
R3 续用自写 GATT 与归档评估 D2（选 Nordic）口径相左 已入 Deviations 待确认；推荐理由：自写版已真机验证，P0 增量小于库迁移+全量回归
R4 ROM 保活矩阵不可控（MIUI/华为行为随版本漂移） 厂商引导 best-effort try-catch；被杀自动收尾作兜底承诺明示用户；真机专项验证

7. Explainer（给零上下文读者）

这次改造在解决什么？ 把"一个连 S7 手表的采集 App"变成"一个换协议就能换项目的采集框架"。类比操作系统的驱动模型：框架是内核（扫描、连接、会话、落盘、导出，永远不动），每种设备协议是一个"驱动包"。今天的问题是驱动模型只建了一半——注册表存在，但主角 B2A 没注册，识别、订阅、UI 全走后门硬编码。

为什么核心动作是"把协议知识做成数据"？ 协议包里有两类东西：数据（怎么认设备的广播特征、订哪些通道、能产出哪些数据流、支持哪些操作）和行为（字节怎么解析）。过去两者混在代码接口里，导致三个消费方拿不到数据：扫描器没法把识别特征交给系统做硬件过滤（于是被节流/息屏扫不到）、UI 没法按能力隐藏不支持的按钮（于是硬编码枚举）、存储没法按协议声明生成 CSV 列（于是闭合枚举锁死词汇表）。B-lite 就是把数据抽成"数据核"，三个消费方全部改喝这口井——行为只剩解析器工厂和 DeviceOps 两个插槽。

为什么文件格式要一次性破格 v2？ 因为有四个"格式一旦带着数据跑起来就改不动"的缺陷叠在一起：多设备混流（数据无法归属）、主键会漂移（事后编辑改文件夹名）、行格式缺路由信息（多通道无法重放）、无版本锚（协议升级后旧数据被静默错解）。趁真实 B2A 数据还没批量产生，一次补齐成本最低。

边界在哪？ D6 划清了层：包内时间语义（样本时间戳/seq/间隔）是采集协议的事，App 不介入——传输层只给包打接收时间标签、raw hex 原样落盘，不反推间隔、不校准设备钟、不排序包内样本。协议若带设备时间戳，解析器解进 device_ts 列；若不带，跨设备毫秒级对齐这类能力就受限。这不是 App 的技术债，而是协议设计要不要承担的取舍，M7 定协议时权衡。

9. Quiz（自测 8 题）

  1. 换一个新协议，理想态改哪些文件？现状差在哪三面？
答案理想态：新增一个协议包 + DI 注册一行。现状差：扫描识别（AndroidBleClient 硬编码判定）、订阅选择（硬编码 B2A/HRS 特征）、UI 门控（VM 直连 B2aDetect、Compose 直用 S7 常量）三面旁路。
  1. 为什么说 DecodedStream 闭合枚举比 UUID 硬编码更深的协议锁？
答案解析器可插拔只解决"字节→事件"，但"事件能说什么词"（有哪些数据流）写死在 domain 枚举——新传感器流要连改 domain/存储/UI/manifest 四层。B-lite 用协议声明的流表 StreamDescriptor 开放词汇表。
  1. startScan(null, LOW_LATENCY) 为什么是最坏组合？
答案无过滤扫描在 Android 8.1+ 息屏直接零结果、连续 30 分钟降级 opportunistic、还最容易触发 5 次/30s 静默节流；带服务 UUID 的 ScanFilter 可硬件 offload，三个问题同时缓解。另注意错误码 6 才是 SCANNING_TOO_FREQUENTLY，5 是资源不足。
  1. 断链后席位怎么走？数据上怎么体现？
答案D5：断链→回连中（保留席位，autoConnect 托管），当前分段关闭并记 gap；回连成功开新分段；超时/手动才释放席位。manifest.devices[i].segments 逐段记录起止与原因。
  1. 为什么 manifest 要加 sessionUuid？现状主键是什么、坏在哪？
答案现状 sessionId=文件夹名，事后编辑采集人/场景会重命名文件夹=主键漂移，导出副本/服务器去重全无稳定键。sessionUuid 创建时生成、永不变，folderName 降级为展示。
  1. D6 的职责边界是什么？包内时间戳归谁？
答案包内时间语义（样本时间戳/seq/间隔）归采集协议定义，App 不介入。传输层只给包打接收时间标签 receivedAtMs + raw hex 原样落盘；解析层按协议把设备时间戳解进 device_ts 列，协议没定义则空。App 不反推间隔、不校准设备钟、不排序包内样本。跨设备对齐能力取决于协议带不带时间戳，是协议设计决策，后果随协议走。
  1. OTA 为什么不能现在"加个命令"就做？
答案三个地基缺口：宿主错（S7Console 挂 viewModelScope 切屏即断，D4 定 DeviceOps+应用级宿主）、WWR 写无流控（写结果不消费，连发大块必丢，P0 补写队列）、未请求 CONNECTION_PRIORITY_HIGH。另有 GATT 服务表缓存问题（OTA 改布局后旧 handle 不失效）。
  1. 真正的丢包计数口在哪一级？为什么现在的 droppedPackets 是假安全感？
答案在 AndroidBleClient 的 MutableSharedFlow(256, DROP_OLDEST)——tryEmit 在该语义下恒返回 true，溢出静默丢弃无计数；manifest 里计的是下游 Channel.trySend 失败，而落盘卡顿时先满的是上游。P2 给上游补计数并按设备入 quality。
