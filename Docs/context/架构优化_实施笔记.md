# 架构优化线 实施笔记（implementation-notes）

> 任务线：通用采集框架化（协议注册中间件 / UI 能力门控 / 保活与长连接 / 数据格式定版）。
> 起于 2026-07-07 架构分析会话；勘察底稿 = 10 智能体工作流（7 子系统 Map + 3 视角 Critique）。
> 本文按用户流程维护四节：Decisions / Deviations / Edge Cases / Open Questions。
> 讨论收尾产出：盲区清单 + 方案对比 + 实现计划的 HTML 总结文档（待出）。

## Decisions（已拍板）

- **D1（2026-07-07）M9「服务器获取解析器」验收档位 = 发版接新协议**。
  新协议随 App 版本发布；服务器只做分发/版本管理/远程开关，不做协议描述热下发。
  推论：方向 C（描述式 DSL + 解释器）出局；Profile 数据核（可序列化 ProtocolDescriptor）仍值得做（服务器可下发配置级更新、网关可复用），但解析行为永远编译期内置。

- **D2（2026-07-07）多 DUT 同采 = 真需求，存储按设备分文件**。
  保留 ≤3 DUT + 1 参考同采；hexlog/CSV 按 deviceId 分文件、DecodedSample 加 deviceId、
  manifest 按设备挂文件清单、多设备命名规则补齐——全部并入文件格式 v2 一次性破格（配合 D1 后续的 sessionUuid/hash/版本锚）。

- **D3（2026-07-07）设备身份 = 固件承诺 static address，App 维持 MAC 锚**。
  App 身份层不动（ScannedDevice.id=MAC 保持）；「广播地址必须为 static（不开 BLE privacy/RPA）」写入固件侧协议冻结项，与 M7 一起对固件排期。
  残余风险入 Edge Cases：第三方参考设备地址策略不受控；未来认证/上市若强制 privacy 则此决策返工（届时走 SN 优先或 bonding）。

- **D4（2026-07-07）控制面 = 独立 DeviceOps 语义层**。
  框架定义语义化操作接口（syncTime/pullLog/reboot/otaTransfer…），协议各自实现并声明支持集；UI 只见接口，不支持的操作不渲染（与能力门控同一套机制）。
  推论：S7Console 改造为 S7 的 DeviceOps 实现；Compose 层不再出现 S7.CTRL_* 常量；OTA 宿主下沉应用级/前台服务（脱离 viewModelScope）。ProtocolProfile 保持采集面干净，不并入控制面。

- **D5（2026-07-07）长连接验收 = 息屏长采 + 断链自动回连**。
  验收：息屏装兜连续采集数小时不断；断链保留席位、autoConnect 托管回连、会话内 gap 标记分段。
  推论：① ConnectionRegistry 语义改「断链≠摘除」，引入席位状态机（CONNECTED/RECONNECTING/FORFEITED）；② 会话状态机支持设备级分段（attach/detach/re-attach 事件 + manifest 段记录）；③ FGS 类型补 location（GPS 流息屏不断）；④ 采集会话期间评估 PARTIAL_WAKE_LOCK；⑤ ROM 保活专项（自启动/上划保护/厂商引导页 + onTaskRemoved）升为必做；⑥ requestConnectionPriority 按场景（OTA=HIGH）。

- **D6（2026-07-07 拍板 + 当日澄清职责边界）时间分层：包内时间语义归协议，App 只打接收标签**。
  **职责划分**（用户澄清，纠正此前把 D6 当"App 风险缺口 + App 侧缓解"的越界表述）：
  · **传输层（App）**：给每个收到的包打接收时间标签 `receivedAtMs`（现状 System.currentTimeMillis, `AndroidPlatform.kt:13`）+ raw hex 原样落盘（source of truth）。**不反推样本间隔、不校准设备钟-手机钟、不排序包内样本**——那是协议/固件的层。是否加单调钟仅是接收标签实现细节, 不作为弥补协议时间语义的手段。
  · **解析层**：按协议把包内时间戳（若协议定义了）解进 `DecodedSample.deviceTsUs` / CSV device_ts 列；协议未定义则该列为空。App 不为 deviceTsUs 造值。
  · **协议层**：采集帧是否带设备时间戳/seq 是协议设计决策（M7 在 B2A/zqdata 主体上定）。其数据质量后果（跨设备对齐、样本间隔、丢包 seq 核算能力）随协议走，**不记为 App 技术债**。
  App 本分内仍做（非"补时间"，是对自身缓冲的诚实）：SharedFlow/Channel 溢出计数、malformed 计数入会话 quality（盲区 D4）。

- **D7（2026-07-07）方案方向 = B-lite（数据核先行）**。
  ProtocolProfile 拆两层：数据核（识别特征 ScanSpec/通道表/流表 StreamDescriptor/能力集，普通 data class，暂不要求可序列化）+ 解析工厂（行为）；扫描、订阅、UI 门控、存储列全部消费数据核；DecodedStream 闭合枚举开放为协议声明的流表。不做 @Serializable/registry_hash 层（服务器线立项时再升级，D1 已定发版制，升级路径保留）。

- **D8（2026-07-07）受试者信息 = 维持明文，内部工具接受**。
  manifest v2 的 subject 字段形状不变（性别/出生/身高体重明文）；风险自担记录在案。**WLAN 上传线（M9）开工前必须重议一次**——届时最低限度做传输加密与受试者代号化评估。

## Deviations（偏离原计划）

- **文档勘误（2026-07-07，codex 交叉验证发现）**：盲区 A2 原写"DecodedStream/CollectType 7 值枚举"混了两个枚举——`CollectType` 实为 **6** 个采集开关（不含 HR，`Collect.kt:9-24`），`DecodedStream` 才是 **7** 值（含 HR，来自参考心率带，`Collect.kt:31-48`）。已就地改正 `设计/架构优化_通用采集框架化.md` §3 A2。结论方向不变。

## 外部验证：codex 交叉核对（2026-07-07）

codex-cli 0.142.0 只读沙箱独立读真实代码，核 13 条承重断言：**11 CONFIRMED / 2 PARTIAL / 0 REFUTED**，无一被推翻，计划事实地基稳。两条 PARTIAL 及三个实施增量：

- **PARTIAL ①（第 5 条）**：枚举混淆——见上 Deviations 勘误。
- **PARTIAL ②（第 12 条）**："全仓无 WakeLock/bonding/priority"字面过头——**生产代码**（.kt/.kts/.xml/.toml）确无 `newWakeLock/createBond/BOND_/requestConnectionPriority`，但 Docs 历史/计划文本有命中。口径应为"生产代码无调用"。
- **措辞收敛（codex A 点）**："注册表空转"须限定为 **B2A 主线空转**；HRS/Mock 解码链路生产代码实际走 `RegistrySampleDecoder.byId()`，并非空转。
- **实施增量①（P1，ScanFilter 可行性 = 好消息）**：Android `ScanFilter.Builder.setServiceUuid(ParcelUuid)` 支持 128 位 UUID——B2A 的 ScanSpec 可真正转成系统硬件过滤（X1 息屏/抗节流对策成立），**但必须用完整 128 位 `0000FFE0-3C17-D293-8E48-14FE2E4DA212`（`B2aDetect.kt:17-23`），不能用 SIG base**。
- **实施增量②（P2，per-device 分文件触点清单）**：不是改文件名的小事——raw writer 以 `DeviceKind` 为 key（`SessionRecorder.kt:38,56-60`）、CSV writer 以 `DecodedStream` 为 key（`SessionRecorder.kt:39,89-92`）、`DecodedSample` 无 deviceId（`DecodedSample.kt:9-14`）。P2 须动 sample+decoder+recorder+layout+manifest+测试六处。
- **实施增量③（P0，写队列最高风险）**：`BleClient.write()` 返回 `Unit`、`AndroidBleClient` 只做 WWR 立即写不消费写回调（`BleClient.kt:71-76`、`AndroidBleClient.kt:289-304`）。P0 改写队列必须**接口契约 + Mock 行为 + 测试一起改**，否则只是把可靠性问题换位置（已并入 §6.6 P0 验收门）。
- P0 整体可行性：codex 判定无硬阻塞；风险梯度 = 写队列（高）> scan(specs) 接口波及 2 VM+Mock（中）> characteristicId 透传/connectionPriority（低）。

## Edge Cases（讨论中确认的边界）

- （随 D3）第三方参考设备（心率带）若用 RPA：MAC 身份跨会话不稳定——参考设备只有 1 台且数据按 kind 归档，影响有限，接受；同产品线 MAC 后四位碰撞：两台 DUT 会话夹名靠时间戳区分，接受。

## 勘察结论速查（详见收尾 HTML 文档）

- 注册表（02 R1–R3）对主协议线空转：无 B2aProfile，识别/订阅/门控三面硬编码（B2aDetect、AndroidBleClient:119/180、4 个 VM 直连 shared.s7）。
- 真正的协议锁 = DecodedStream/CollectType 闭合枚举（数据词汇表），不是解析器。
- 扫描现状 startScan(null, LOW_LATENCY) 为最坏组合：8.1+ 息屏零结果、30min 降级、5/30s 静默节流（代码注释错把 5 当节流码，实为 6）。
- 长连接四件套空白：connectionPriority / 断链发现延迟 / autoConnect 托管回连 / FGS 类型（只传 CONNECTED_DEVICE；A15 dataSync 6h 配额影响 M9 上传）。
- 数据层七雷：多 DUT 混流、时间轴无权威（全墙钟+deviceTsUs 冒充）、MAC 身份无 RPA 后路、droppedPackets 计数点错位（SharedFlow DROP_OLDEST 静默）、会话主键=文件夹名、hexlog 无特征标识（characteristicId 未透传）、profileId 无版本段。
- Nordic 库：D2 拍板已选但代码零依赖（自写裸 GATT）；替换时机应在 M7 传输绑定动工前。
