BlueTrace 架构评估：现状、问题与演进（2026-07-06）

输入：四路代码审查（代码审查报告，波次①–④已修复高/中危主体）+ 本轮结构层审计（模块/包依赖矩阵、KMP 边界、DI、协程纪律逐项核对，file:line 佐证）。 定位：回答"当前架构有什么问题、离最佳实践差在哪、M7/M8 来临前该怎么演进"。方案衔接 02 协议注册架构设计，不另起炉灶。 读者假设：不要求 Android 背景。§0 先讲本文用到的 Android/KMP 机制，问题条目里再逐个展开"为什么这是问题"。 拍板记录（2026-07-06）：D1 ✅ R1–R3 立即落码；D2 ✅ 采集档传输选 Nordic Kotlin-BLE；D3 ✅ 上 GitHub Actions CI。

0. 机制速览（给非 Android 背景的读者）

每条给一个嵌入式/固件侧的类比，帮助建立直觉。类比只取神似，细节以正文为准。

① 协程与派发器（Dispatchers）——"任务跑在哪个线程池上" Kotlin 协程是轻量任务（类比 RTOS task，但由用户态调度、挂起不占线程）。协程必须指定"派发器"决定它在哪类线程上执行：

派发器 线程池 该放什么 类比
Dispatchers.Main UI 单线程 只做界面更新，绝不能阻塞 中断上下文：手快进快出，卡它=界面掉帧/ANR
Dispatchers.Default CPU 核数个线程 纯计算（解析、排序） 算法任务的时间片池
Dispatchers.IO 弹性大池（64+） 阻塞式文件/网络/数据库 专门的后台 flash/通信任务

关键纪律：阻塞式磁盘写必须放 IO 池。 放 Default 池 = 把 flash 写操作塞进算法任务的时间片——写盘卡住时，本该做计算的线程被占死（本文 A1 正是此问题）。

② 结构化并发与异常传播——"没装 fault handler 的系统" 协程组织成父子树（CoroutineScope = 一棵树的根）。子协程抛出未捕获异常时向上传播，在 Android 上最终行为是杀死整个进程（相当于未注册 HardFault_Handler 直接复位）。CoroutineExceptionHandler 是挂在 scope 根部的全局兜底钩子——本项目此前一处都没装（A2）。SupervisorJob 表示"一个子任务失败不连坐其他子任务"，本项目已正确使用。

③ Android 进程生命周期——"随时可能被看门狗复位的板子" Android 为回收内存随时可能杀掉后台进程，用户从"最近任务"回来时系统会用保存的界面状态（savedInstanceState）恢复界面骨架，但内存里的对象全没了。所以任何"内存中的运行态"都要回答：进程被杀后怎么办？本项目的答案是三层：前台服务（提高进程优先级，声明"我在干活别杀我"）+ 会话开口自动收尾（重启后把没写完的会话补结束标记）+ 幽灵页防护（恢复出来的运行页发现没有活动会话就自动退回）。这三层在波次①已修好。

④ Compose 与 remember——"RAM 变量 vs 掉电保持变量" UI 用 Jetpack Compose（声明式：界面 = f(状态)，状态变了框架自动重画）。remember { } 的变量存在内存里，转屏/进程重建就清零（RAM 变量）；rememberSaveable { } 会进 savedInstanceState（类比掉电保持的 NVM 变量）。选错会出"转屏后弹层复活/输入丢失"这类 bug（波次①修过一批）。

⑤ ViewModel 与 MVVM——"屏级状态容器" 每个屏幕配一个 ViewModel（VM），持有该屏状态（StateFlow，类比可订阅的寄存器组），转屏时 VM 存活（比 remember 长命，比进程短命）。约定：界面只读 VM 的状态、只调 VM 的方法；跨屏共享的状态放 DI 单例。本项目 19 处界面代码绕过 VM 直接从 DI 拿领域对象（C2），相当于外设驱动绕过 HAL 直接戳寄存器——能跑，但破坏了分层的可替换性与可测性。

⑥ 依赖注入（Koin）——"运行期的链接表/设备树" DI 容器在 App 启动时按"绑定表"（AppModule）组装对象图：声明 single<BleClient> { AndroidBleClient(...) } 后，任何要 BleClient 的构造函数自动拿到这个实例。类比链接期符号绑定：接口是符号，绑定表决定链到哪个实现——这就是 Mock/真实一行切换的机制。"裸泛型绑定"问题（C1）：把 Path、CoroutineScope 这种通用类型不加名字直接注册，等于把两个不同用途的全局变量都叫 g_buf——第二个注册进来就静默覆盖或注错。

⑦ KMP 源集——"中间件层与板级支持包" Kotlin Multiplatform 把代码分源集：commonMain = 平台无关核心（类比 HAL 之上的中间件，只许用纯 Kotlin 库）；androidMain/iosMain = 各平台实现（类比 BSP）。iOS 二期的成本 ≈ commonMain 的纯度。本项目 commonMain 实测零平台泄漏、平台能力全部通过接口注入（连 expect/actual 这种平台桥接语法都没用），这是 iOS 成本最低的形态——前提是别再把纯逻辑写到 app 模块里去（B3 正是清这笔债）。

⑧ BLE GATT 单飞约束——"半双工总线" Android 蓝牙栈同一时刻只允许一个在途 GATT 操作（读/写/订阅），上一个没回调完成就发下一个会静默失败。正确做法是操作队列串行化（类比半双工总线的收发互斥）。Nordic Kotlin-BLE 库的核心价值就是把队列、重连、MTU 协商这些"总线纪律"封装掉（D2 选它的理由）。

1. 一句话结论

骨架是对的，值得保持；问题集中在四处：一个真正的线程模型错误（采集落盘跑在 CPU 池）、两个包级依赖环（Mock 混放接口包）、一处状态双真相（ConnectionRegistry vs linkState）、一批"该在 shared 却写在 app"的 iOS 债。 M7 的协议架构不必新设计——02 号注册式设计成立，且 R1–R3 不等协议冻结即可落码。

4. 问题清单（按优先级，含背景展开）

P0 · 线程与健壮性（立即修，约半天）

A1 · 采集落盘跑在 Dispatchers.Default（CPU 池）

  • 背景：见 §0-①。会话的所有事件处理（含 okio 阻塞写盘）跑在 runScope 上，而 runScope 继承了应用级 scope 的派发器 = Dispatchers.Default（AppModule.kt:53 + DefaultSessionController.kt:125-140）。
  • 为什么是问题：Default 池只有 CPU 核数个线程，是给纯计算用的。写盘是阻塞操作——盘一慢（低端机、EMMC 繁忙），计算线程被占死；真实 BLE 数百包/秒时，写延迟会放大、其他计算任务被饿。Mock 每秒 ~50 包所以从没暴露。
  • 改法：给会话注入专用 IO 派发器 Dispatchers.IO.limitedParallelism(1)——放进 IO 弹性池（阻塞合法），limitedParallelism(1) 保证仍是单线程执行，不破坏"单消费者串行无竞态"的既有语义。
  • 影响面：AppModule 一条 named 绑定 + DefaultSessionController 一个构造参数（默认空上下文，测试零改动）。

A2 · 全仓零 CoroutineExceptionHandler

  • 背景：见 §0-②。应用级 scope（AppModule.kt:53）与日志写线程 scope（:68）都没挂异常兜底钩子。
  • 为什么是问题：任何跑在这两个 scope 上、且自身没 try/catch 的协程，一抛异常就是整个 App 崩溃。波次①给采集消费循环装了局部兜底，但这只是众多协程之一；导出任务（已挪到应用级 scope）、未来的新任务都在裸奔。
  • 改法：两个 scope 各挂 handler：应用级的记入 DiagnosticsLog（用户可在"应用日志"里看到 ERROR），logWriter 的写 logcat（不能写自己，会递归）。
  • 影响面：AppModule 两处绑定内改，其余零改动。

A3 · IO 边界不统一（线程切换责任散落在调用方）

  • 背景：SessionStore（会话列表/详情/删除）和 DeviceLogStore（设备日志存取）的方法是阻塞式的——它们自己不管线程，谁调用谁负责先切到 IO 池。于是 withContext(Dispatchers.IO) 这句"切线程"的样板散落在 7 个调用点，其中 3 处直接写在界面代码（Composable）里（ConsoleLogListScreen.kt:46 等）；而采集主页"开始采集"路径干脆忘了切——在主线程建目录写 manifest（审查架构#4，低端机上点按钮会掉帧）。
  • 为什么是问题：把线程纪律交给每个调用方 = 靠人肉记忆守规矩，漏一处就是主线程卡顿。最佳实践是仓库层自守线程：数据类的公开方法自己声明 suspend 并在内部切 IO，调用方无脑调用即可（本项目 SqlDelightSubjectRepository 已是此模式，照抄即可）。
  • 改法：两个 Store 的公开方法改 suspend + 内部 withContext(注入的 io)；删光调用点的散落切换；start() 路径的建目录/写 manifest 挪进会话 IO 协程（与 A1 联动）。
  • 影响面：约 10 个调用点签名跟随（机械改动）；commonTest 的 SessionStoreTest 7 例包一层 runTest。

P1 · 结构边界（M7 开工前修，约一天）

B1 · 包级双向依赖环 ×2

  • 背景：包（package）之间的依赖应当是单向的（类比模块分层：驱动层不该反过来 include 应用层头文件）。审计的 import 矩阵发现两个双向环：ble↔protocol、ble↔s7。
  • 成因：都是同一个——Mock 实现和接口混放在 ble 包。MockBleClient（实现）要 import protocol 的编码器和 s7 的模拟手表来造假数据，而 protocol/s7 又正当地依赖 ble 的接口类型。接口无辜，实现拖累。
  • 为什么是问题：有环的包无法单独理解、单独替换、单独测试；将来拆模块（如果需要）直接卡死；也是"第三种协议接入即塌"的前兆信号。
  • 改法：Mock 实现迁入 ble.mock 子包。依赖变成 mock→{ble,protocol,s7} 单向扇出，环消失。顺带为 02 设计的 R2（Mock Profile 化）铺路。
  • 影响面：移文件 + 全仓 import 批改（机械）。

B2 · 已连接状态"双真相"

  • 背景：谁连着？有两个答案：ConnectionRegistry（一个手工维护的已连设备列表，采集主页/限额校验用）和 BleClient.linkState（BLE 层的真实连接状态流）。两者没有机制性关联——每个 ViewModel 在 connect/disconnect 时手动调 registry.add/remove 对齐（ConsoleConnectViewModel.kt:137-142 等）。
  • 为什么是问题：靠调用纪律维持的一致性必然漂移——哪天设备自己断了（超距、没电）而没人调 remove，Registry 里就留一个幽灵设备，采集主页显示"已连 1 台"但实际没有。这类 bug 极难排查。
  • 改法：单一事实源。Registry 下沉 shared 并改为事件驱动：自己订阅 linkState，见 DISCONNECTED 自动清退；VM 只管发起 connect/disconnect，不再碰列表。
  • 影响面：Registry 移包 + 增订阅逻辑；各 VM 删手动 add/remove（净删代码）。

B3 · iOS 债：纯 Kotlin 逻辑滞留在 app 模块

  • 背景：见 §0-⑦。app 模块是 Android 专属的——写在这里的代码 iOS 全部要重写。审计发现一批零 Android 依赖的纯逻辑住错了地方：ConnectionRegistry(41 行)、CollectDraft(31 行，场景草稿)、Subject→S7Person 映射（写手表用户信息的域转换）、refreshAll 聚合编排（逐项读设备信息、单项失败不阻断）。另有半债：MediaStoreExporter 里 zip 组包/勾选过滤是纯逻辑，只有"落到 Download 目录"那一下是平台的。
  • 为什么现在做：现在下沉 = 搬文件 + 改 import；M8 时再下沉 = 从 1300 行 ViewModel 里做代码考古。
  • 改法：四块纯逻辑下沉 commonMain（Registry 随 B2）；Exporter 拆 SessionZipPacker（commonMain）+ 平台落盘接口。DeviceLogStore 顺手从 domain 包挪到 data.android（包名与职责对齐，C4）。

B4 · 协议接入仍是全局单解码器 → 落地 02 设计 R1–R3【D1 已拍板 ✅】

  • 背景：现在全 App 只有一个 SampleDecoder 单例，靠 DeviceKind（DUT/参考）二分决定怎么解析。接第三种协议（S7 采集、UHTP）时这个二分直接塌掉。
  • 方案：02 号注册式协议架构已给出完整设计——ProtocolProfile（一个设备族的接入包：怎么识别设备、订阅哪些通道、怎么解析、怎么编码命令）注册进 ProtocolRegistry，按广播特征自动匹配；解析产物统一为 ProtocolEvent 事件模型（样本/回执/状态/文件块/坏帧）。新协议接入 = 写一个 Profile 类 + 注册一行。
  • 节拍：R1 骨架+适配器（上层零改动）→ R2 Mock Profile 化（删全局单例）→ R3 事件模型接入编排（22 单测护航）。三步全是 shared 内重构，不等协议冻结；R4（HRS 心率带真实链路首连）只等硬件；R5（V0/UHTP 解析器 = M7 交付物）等冻结，但骨架就位后冻结不再阻塞架构。
