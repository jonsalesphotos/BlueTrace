02可注册通道解析器架构 —— ProtocolProfile / ChannelParser / Registry

完整设计见 02_parser_registry_design.md。解决三件事：S7 等新协议接入、通道（GATT UUID）成为一等公民、多协议同会话并存。新协议接入 = 写一个 Profile + 注册一行。

BleClient.notifications(deviceId) BleNotification(deviceId, channel, bytes, receivedAtMs) │ ▼ ProtocolRegistry.resolve(device) ──► DeviceParserHost（每设备；重连 reset） │ 按 channel 取 profile.createParser(ch) ├─ BlueTraceV0Parser FrameReader(hdrCrc8切帧) → Reassembler(msgId分片重组) → dispatch(msgType) ├─ S7QualityParser S7 自有成帧/校验 → S7 消息表（待 S7 侧帧格式） └─ HrsParser 标准 0x180D/0x2A37 心率解析（Polar H10 参考） │ ▼ List<ProtocolEvent> Samples / CommandAck / Capability / State / AlgoResult / FileChunk / Malformed DefaultSessionController（单 Channel 串行，机制不变） ├─ Samples → SessionRecorder（hexlog+CSV 现路径） ├─ FileChunk → OfflineImporter（03） └─ Ack/Capability/State → ControlPlane（04 手表控制） └─ Malformed → 质量小结 + 诊断日志

核心接口（commonMain）

ProtocolProfile：id · matches(device) · notifyChannels · writeChannel · createParser(ch) · commandEncoder()
ChannelParser：parse(notification) → 事件列表；reset()（重连清分片缓冲）
ProtocolRegistry：DI 期组装、运行期只读；session_manifest.json 记录命中的 profile.id 供离线重放选解析器

迁移路线（不打断 Mock）

R1 RegistrySampleDecoder 适配器包住注册表——上层零改动 → R2 Mock 协议入注册表 → R3 SessionController 事件化 → R4 HrsProfile + Nordic 真实链路首连（风险前置，不等协议冻结） → R5 BlueTraceV0Parser（=M7 交付物）与 S7QualityParser 各自独立落地

S7 接入待办清单（提供给 S7 侧）：Service/Notify/Write UUID · 广播识别特征 · 帧格式（同步字/长度域/校验算法/字节序）· opcode 消息表与单位缩放 · 有无分包与命令回执模型 · 一段真实抓包 hexlog（直接进 commonTest 回放）。

03采集链路协议 BTCP/1 —— 在线流式 + 离线回传

完整设计见 03_collect_protocol_design.md，schema 草案 btcp1_draft.proto。信封完全复用 SPEC §4（12B TLV 头 + 分片 + CRC），对 bluetrace_v0.proto 纯增量扩展。

核心决策 P-1：离线会话文件 = 在线帧流的落盘镜像。设备把与在线 Notify 同构的「12B 头+payload 帧序列」写进 Flash；App 导入时逐帧喂同一个 BlueTraceV0Parser。双端都只有一套编解码，在线/离线数据进同一条 hexlog→CSV→manifest 落盘流水线。

帧信封（继承 v0.1，不改）

// 一帧 = 12B 头 + payloadLen 字节 protobuf；T=msgType L=payloadLen V=protobuf
ver(1) msgType(1) flags(1) fragIndex(1) fragCount(1) hdrCrc8(1) pktSeq(2LE) msgId(2LE) payloadLen(2LE)
// 分片：偶发大消息（目录页等）；高吞吐路径（批包/离线块）自控大小禁分片（D-5：App 不配 MTU）

msgType 增量（0x50–0x5F 段划给离线回传）

msgType消息方向用途
0x13TimeSyncPong设→AppNTP 式对时应答（四时间戳，offset=((t1−t0)+(t2−t3))/2；锚点写 manifest 并持久化到设备）
0x50SessionCatalog设→App已存会话目录（分页；允许标准分片）
0x51/0x52/0x53TransferStart / Chunk / End设→App块传输：元数据+全文件 CRC32 → 数据块（单包禁分片）→ 完成/中止
0x54StorageInfo设→App设备存储占用（控制台复用）
0x55OfflineSessionHeader仅文件内离线文件首帧：会话元数据 + 对时锚点对（device_ts↔UTC）
0x40 oneof +8ListSessions / OpenTransfer / TransferCredit / CloseTransfer / DeleteSession / QueryStorage / TimeSyncPing / FormatStorageApp→设命令面增量（tag 9–16）；SessionControl 加 app_session_id / wall_clock_ms / offline 字段

离线回传时序（credit 滑动窗口流控）

App 设备 │ ListSessions(page) │ │ ◄──────────── 0x50 SessionCatalog ── │ 离线A · 已存会话列表 │ OpenTransfer(id, resume_off, maxChunk=201) │ │ ◄─ Ack ─ 0x51 TransferStart(chunk, total, CRC32) │ │ TransferCredit(16) ────────────────► │ 授信 16 块 │ ◄─ 0x52 Chunk(off=0) ×16 连发 ────── │ 离线B · 导入中（进度=offset/total） │ 每消费 8 块补 TransferCredit(8) ────► │ 滑动窗口流水线 │ ◄─────────── 0x53 TransferEnd ────── │ │ CRC32 校验通过 → DeleteSession ─────► │ （用户确认后显式删） │ 帧流重放进 BlueTraceV0Parser → 落盘 │ 离线C · 分配场景/用户 → manifest(origin=offline)
  • 断点续传：重连后 OpenTransfer(resume_offset)；掉电最多损尾帧（append-only），解析容忍截断；
  • 完整性三层：BLE 链路层 CRC → 帧头 hdrCrc8 → 全文件 CRC32；
  • 吞吐估算：保守 ~27 kB/s、理想 60–100 kB/s → 1h IMU 6轴@100Hz(S16)≈4.3 MB，1.5–3 分钟拉完；
  • IMU 在线路径：ACC/GYRO/MAG 各一条 HighFreqBatch 流（channel_count=3、SF_S16、batch_seq 独立），SetPacking(25, 250ms) 单包不分片，带宽仅 ~1.2 kB/s；
  • 开放问题 O-1…O-6（传输时在线面暂停？存储满环形覆盖？…）已列清单待固件端拍板。

04手表控制界面规划 —— 设备控制台（KMP 就绪）

完整规划见 04_watch_control_plan.md。入口 = 设置 Tab 既有占位「设备维护（DUT）」；协议原语（Command/Ack + Capability/State）v0.1 已备。

界面（沿 V4 设计语言的设置子页）

设备卡（固件/RSSI/连接态）→ 对时状态 + 立即对时 → 存储占用条（复用 StorageBreakdown）→ 传感器区：能力表驱动（开关 + supported_rates 下拉 + 实际生效值对账，被夹紧时琥珀标注）→ 高级区（SetPacking / 算法多选，折叠）→ 危险区（FormatStorage 红色二次确认）。 采集运行中进入 = 只读锁定。

分层（逻辑全下沉 shared commonMain）

DeviceCommandBus：串行下发 · cmd_id↔Ack 配对 · 3s 超时；
DeviceControlSession：capability/state/storage/timeSync 的 StateFlow 快照 + 乐观更新/回滚/对账；
Compose 只留薄壳（iOS SwiftUI 后期对接同一控制器）；commonTest 用 Fake bus 覆盖超时/乱序 Ack/夹紧对账。
