# BlueTrace UI 产品与交互说明

> 本文档面向产品、业务、测试和设计评审，说明 BlueTrace Android App 一期页面范围、核心用户流程、异常处理和体验规则。
>
> 技术模块拆分、状态模型、前台服务、后台任务、BLE 接入顺序见 [BlueTrace_UI_Implementation.md](BlueTrace_UI_Implementation.md)。

---

## 1. 产品目标

BlueTrace 是一个面向 BLE 数据采集的移动 App。一期目标是优先完成稳定的采集闭环。

用户需要完成的核心事情是：

```text
打开 App
→ 确认手机环境可用
→ 选择本次采集使用的设备
→ 开始采集
→ 采集中看到状态和实时数据
→ 停止采集
→ 查看本次结果
→ 导出或管理历史文件
```

一期不追求完整覆盖所有高级设置，优先保证：用户能完成采集、理解采集状态、安全停止任务，并能找到和导出数据文件。

---

## 2. 一期页面清单

一期建议先做 5 个主要页面，加几个关键弹层。

| 页面 / 入口 | 用户看到的名字 | 主要作用 | 一期优先级 |
| --- | --- | --- | --- |
| 权限检查页 | 权限与环境检查 | 检查蓝牙、定位、BLE 权限、通知权限等 | 必做 |
| 设备选择页 | 设备 | 选择本次采集要用的 DUT / REFERENCE 设备 | 必做 |
| 设备扫描页 | 扫描设备 | 扫描附近 BLE 设备并连接 | 必做 |
| 数据采集页 | 数据采集 | 显示采集状态、实时数据、停止按钮 | 必做 |
| 历史文件页 | 文件 / 历史记录 | 查看已采集文件，导出或删除 | 建议一期做 |
| 开始前提醒弹层 | 开始前检查 | 开始采集前提示非阻断问题，例如通知权限未开 | 必做 |
| 停止确认弹层 | 停止采集确认 | 避免用户误触停止 | 必做 |
| 采集恢复通知 | 恢复采集 | App 被系统杀掉或重启后，引导用户回到 App | 后续增强 |

页面职责简表：

```text
权限页：判断当前环境是否允许采集
设备页：配置本次采集使用的设备角色
采集页：展示运行状态、实时数据和停止入口
文件页：承接采集结果查看、导出和删除
```

## 3. 页面关系总览

正常用户流程：

```text
启动 App
  ↓
权限与环境检查
  ↓
设备选择
  ↓
扫描并连接设备
  ↓
回到设备选择页确认
  ↓
开始采集
  ↓
数据采集页
  ↓
停止采集
  ↓
采集结束摘要
  ↓
查看 / 导出文件
```

如果用户已经有历史文件，可以从首页或底部 Tab 进入文件页查看。

如果采集中用户切到其他页面，App 应该仍然能明显提示：

```text
正在采集 · DUT + REFERENCE · 00:12:43    [打开采集页]
```

该提示用于避免用户在离开采集页后失去对运行中任务的感知。

## 4. 启动 App 后发生什么

用户打开 App 后，系统先判断两件事：

1. 手机环境是否满足采集条件
2. 之前是否有未结束或可恢复的采集任务

可以理解为：

```text
打开 App
  ↓
检查蓝牙 / 定位 / BLE 权限
  ↓
检查是否有上次未完成的采集
  ↓
根据结果进入对应页面
```

常见结果：

| 情况 | App 应该怎么做 |
| --- | --- |
| 蓝牙没开或关键权限缺失 | 进入权限检查页，引导用户处理 |
| 环境正常，且没有正在恢复的采集 | 进入设备选择页 |
| 之前有未完成采集，且能恢复 | 直接回到数据采集页，并提示“已恢复上次采集” |
| 之前有采集，但部分设备需要重新连接 | 进入采集页，同时提示哪个设备需要重新扫描 |
| 之前有采集，但无法自动恢复 | 回到设备选择页，预填上次设备信息，让用户重新扫描 |

## 5. 权限与环境检查

这个页面的作用是让用户知道：手机现在是否具备采集条件。

重点检查：

| 检查项 | 为什么需要 |
| --- | --- |
| 蓝牙开关 | 没有蓝牙无法扫描和连接设备 |
| 定位服务 | Android 上 BLE 扫描通常依赖定位服务 |
| BLE 扫描 / 连接权限 | Android 运行时权限要求 |
| 前台服务通知权限 | 采集在后台继续运行时，需要通知栏提示 |
| GNSS 定位权限 | 如果要把手机位置写入采集记录，就需要 |

权限分成两类：

| 类型 | 处理方式 |
| --- | --- |
| 硬性条件 | 不满足就不能开始采集，例如蓝牙关闭、BLE 权限缺失 |
| 建议条件 | 不满足也能继续，但会影响体验，例如通知权限未开 |

产品规则：

```text
关键权限必须阻断采集
建议权限可以提醒，但不应阻断全部流程
```

## 6. 设备选择页

设备选择页是开始采集前的主页面。

用户在这里决定本次采集要用哪些设备。

常见模式：

| 模式 | 说明 |
| --- | --- |
| 只采 DUT | 只连接待测设备 |
| DUT + REFERENCE | 同时连接待测设备和参考设备，例如心率带 |

页面上应该显示：

- 当前已选设备
- 每个设备的连接状态
- 是否可以开始采集
- 进入扫描页的按钮
- 开始采集按钮
- 进入历史文件的入口

开始采集按钮的规则：

```text
至少有 1 个有效设备连接成功
并且关键权限已满足
才允许开始采集
```

---

## 7. 设备扫描页

设备扫描页用于寻找附近的 BLE 设备。

用户选择某个角色后进入扫描页，例如：

```text
为 DUT 选择设备
或
为 REFERENCE 选择设备
```

页面需要提供：

- 正在扫描状态
- 设备名称
- 设备信号强度
- 设备类型或服务信息
- 连接按钮
- 无设备时的提示
- 蓝牙关闭或定位异常时的提示

建议支持简单过滤：

| 功能 | 作用 |
| --- | --- |
| RSSI 阈值 | 过滤太远或信号太弱的设备 |
| 名称搜索 | 快速找到指定设备 |
| 分组展示 | DUT 和 REFERENCE 分开看，减少误选 |

连接成功后，用户回到设备选择页。

---

## 8. 数据采集页

数据采集页是 App 的核心运行页面。

采集中用户需要一眼看到：

- 是否正在采集
- 已采集时长
- DUT / REFERENCE 是否在线
- 数据是否在持续进入
- 当前文件大小
- 是否有断线、写入失败等异常
- 如何安全停止采集

页面可以先提供两种展示方式：

| 模式 | 说明 | 一期建议 |
| --- | --- | --- |
| 文本流模式 | 显示原始 Hex 或日志流 | 优先做 |
| 图表模式 | 显示实时曲线 | 可以先做简单版或二期 |

停止采集需要明确的防误触机制。

推荐交互：

```text
点击 End
→ 弹出停止确认
→ 用户按住 2 秒
→ 松手则取消
→ 按满 2 秒才真正停止
```

该交互用于降低采集中误触停止造成数据中断的风险。

---

## 9. 采集运行中如何处理页面切换

采集开始后，它不再是普通页面操作，而是一个持续运行任务。

用户可能会：

- 切到文件页
- 切到设置页
- 按返回键
- 锁屏
- 切到其他 App

产品规则建议：

```text
采集中不能让用户误以为“返回键 = 停止采集”
采集中需要始终有明显运行提示
真正停止必须经过明确确认
```

如果用户在采集页按返回键，可以弹出选择：

| 选项 | 结果 |
| --- | --- |
| 最小化 | App 退到后台，采集继续 |
| 停止并退出 | 进入停止确认流程 |
| 取消 | 留在采集页 |

如果用户切到其他页面，顶部显示“正在采集”提示条，点击可回到采集页。

---

## 10. 采集结束摘要

采集结束后，不要直接把用户丢回首页。

应该先展示一个结束摘要，让用户确认这次采集是否成功。

摘要内容：

- 采集时长
- 采集设备
- 文件数量
- 文件大小
- 是否有异常
- 保存位置
- 导出按钮
- 查看文件按钮
- 返回首页按钮

示例：

```text
采集已结束
时长：00:12:43
设备：DUT + REFERENCE
文件：1 个 .rawdata
大小：8.63 MB
```

## 11. 文件与历史记录

文件页解决两个问题：

1. 用户采完以后能找到数据
2. 用户能把数据导出给电脑、服务器或分析人员

一期建议支持：

- 按时间展示历史采集
- 查看采集详情
- 导出文件到 Downloads
- 删除文件
- 显示文件大小
- 显示对应用户、设备、采集模式

可以暂缓到二期的功能：

- 批量上传服务器
- 批量下载压缩包
- 编辑文件备注和标签
- 复杂筛选
- 云端同步状态

一期最重要的是：

```text
采完的数据不能丢
用户必须知道文件在哪里
用户必须能导出
```

## 12. 异常与提示

采集类 App 的风险不只在页面流程，也在异常状态是否被清楚表达。

以下异常需要提前定义提示策略。

### 12.1 蓝牙关闭

如果采集中蓝牙被关闭：

```text
顶部红色提示：蓝牙已关闭，采集暂停
按钮：打开蓝牙
```

如果只是某个设备断线：

```text
顶部黄色提示：REFERENCE 已断开，DUT 仍在采集
按钮：重新连接
```

### 12.2 权限被撤销

如果用户去系统设置里撤销关键权限：

```text
停止相关采集能力
提示用户重新授权
必要时回到权限检查页
```

### 12.3 存储失败

如果文件写入失败：

```text
顶部红色提示：存储失败
建议操作：停止采集并导出已有数据 / 清理空间
```

### 12.4 App 被系统杀掉

如果 App 被系统回收、手机重启或进程异常退出：

```text
通知栏显示：BlueTrace 有未完成采集，点击恢复
用户点击后回到 App
App 尝试恢复采集或引导重新连接设备
```

## 13. 一期与二期范围建议

### 一期必须完成

- 权限与环境检查
- 设备扫描与连接
- DUT / REFERENCE 角色选择
- 开始采集
- 数据采集页
- 安全停止采集
- 采集结束摘要
- 文件保存
- 文件查看与导出
- 基础异常提示

### 一期可以简化

- 图表模式可以先做简单曲线，甚至先只做文本流
- 文件页可以先不做复杂筛选和批量操作
- 设置页可以先只放必要配置
- 用户管理可以先做最小字段
- 采集模式可以先用固定预设

### 二期再增强

- 实时图表高级展示
- 服务器上传
- 实时透传
- 日志上传
- 在线星历
- 复杂用户管理
- 批量文件操作
- 文件备注、标签、别名
- 更完整的异常恢复

---

## 14. 关键产品规则

这些规则建议作为后续设计和开发的统一依据。

1. 用户必须知道当前是否正在采集。
2. 采集中不能因为误触返回键而停止。
3. 停止采集必须有明确确认。
4. 至少一个设备连接成功后才允许开始采集。
5. 蓝牙、定位、BLE 权限是关键条件，缺失时要明确提示。
6. 通知权限可以建议开启，但不要在所有场景都强制阻断。
7. 采集结束后必须展示结果摘要。
8. 文件必须能被用户找到和导出。
9. DUT 和 REFERENCE 可以共用一套设备选择流程。
10. 一期优先保证稳定采集，不优先做复杂管理后台。

---

## 15. 给团队的统一理解

BlueTrace 一期不应优先做成配置项很重的工具箱，而应先做成可靠的数据采集 App。

最小可用版本应该让用户完成：

```text
我能打开 App
我知道权限是否正常
我能选设备
我能开始采集
我能看到采集状态
我能安全停止
我能找到文件
我能导出数据
```

只要这条链路稳定，后续再补服务器上传、实时透传、日志、星历、复杂用户管理，都会更稳。
