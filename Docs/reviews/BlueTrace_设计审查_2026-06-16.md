## 0. 一句话结论

**地基非常扎实，但"最新原型"和"文档契约"已经分叉，落地前必须先做一次收敛对齐——否则开发会按两套互相矛盾的信息架构各做各的。** 当前资料的"想清楚"程度（需求、状态机、协议、跨平台）远超一般草案；真正的风险不在缺思考，而在**版本漂移**和**少数已定决策没有回灌到原型/文档**。

## 1. 已经做得好的部分（值得保留，不要动）

1. **文档分层堪称范本**：`PRD（做什么）→ UX Flows（怎么操作）→ Design System + 原型（长什么样）→ Architecture（怎么落地）` 四层互不污染，`REQUIREMENTS.md` 作单一事实源 + 计划控制台，D-1…D-10 决策全部闭环并带变更记录。这是多数团队做不到的。
2. **状态优先的交互建模**：UX Flows 以**状态机**（SessionState / RoleRuntimeState / 传感器路状态）而非"页面"驱动，`supervisorScope` 单路失败隔离的产品规则贯穿 PRD/UX/架构三层，措辞一致。
3. **异常体系是全场最强项**：红/黄/蓝三级 Banner + 异常合屏（屏 17）+ 进程恢复弹层（P5，逐角色判断 DUT 命中 / 参考需重扫）+ 导出完成 Toast——边界情况清单里的 8 个场景原型几乎都画了。
4. **传感器配色系统**：PPG 绿 / ECG 红 / IMU 蓝 / 地磁紫 / 温度橙，固定贯穿配置→波形→分包流→摘要，色 + 文字双编码（满足"色彩不作为唯一信息载体"）。
5. **强状态任务的防误触设计**：采集中"返回 ≠ 停止"、长按 2 秒环形进度结束（屏 13 含临界态），是对采集类工具的正确理解。
6. **跨平台策略成熟**：统一设计语言 + 仅外壳层分叉（导航/弹层/安全区/后台机制），并有组件级映射表。
7. **v4 的视觉精修确实有效**：字阶收紧（标题 19 / 卡标题 14 / 三级字重 700-600-400）、4/8 网格、采集页一页放下，层级靠对比与字重而非字号——方向对。

## 3. 缺失的页面 / 状态 / 组件清单

**缺失页面（若保留 v4 范围）**

- 受试者管理：列表 / 新建 / 编辑（v4 把它做成主入口，却没有这三屏）。
- 独立设置页（若定 v3 为准）。
- "离线收集"的去向页或解释（当前是个无目的地的 CTA；且一期本就无实时透传，"离线 vs 在线"语义不明）。
- 首启引导 / 空状态 Home。

**缺失状态**

- 扫描：无结果（空态）、扫描中→连接中→已连接过渡、连接失败/超时、定位/权限被拒后的内联阻断态。
- 列表：历史为空、数据为空、长文件名溢出、多选选中态/排序（v4 数据页"选择"无选中视觉）。
- 导出：进行中、失败/空间不足预检。
- 恢复链路：设备地址失效→"正在按 Service UUID 重扫…"、协议版本不匹配错误、重连后**设备实际配置 ≠ App 预期**（如申请 100Hz 设备只给 50Hz）的对账提示（架构/协议都依赖它，UI 无表达）。

**缺失组件（需补进 Design System 目录）**

- 底部导航 BottomNav、入口中枢卡 EntryTile、受试者卡、Wear/Unwear 分段、搜索框 Search、文件卡 FileCard、设置列表行（含开关）、Toast、空态 EmptyState。

## 5. 下一步建议使用哪个 design 能力，以及为什么

> 重要前置：**最关键的下一步不是任何"画图"能力，而是一次产品决策**——C-1（哪套 IA）与 C-2（受试者/Wear-Unwear/离线 是否进一期）必须先由人拍板。设计能力都在这之后才有意义。

拍板后，按下列顺序：

1. **`/design-system`（首选，紧接产品决策之后）**
   理由：当前最大的设计债是"原型与设计系统文档分叉、组件目录缺项、token 漂移"（M-2）。design-system 正是用来审计命名/硬编码、补全组件变体与状态、把 v3/v4 的积木收敛成一套权威组件库。它直接修复一致性与"缺失组件"，是后续一切的地基。

2. **`/design:ux-copy`（补状态文案）**
   理由：第 3 节缺的大量空态/错误态/恢复态，文案是其骨架（空结果说什么、连接失败给什么动作、协议不匹配怎么解释）。在补画这些状态前先定文案，能避免返工。

3. **`/design:accessibility-review`（冻结交付前）**
   理由：L-3 的动态字号 / 触控目标 / 对比度 / "一页不滚动"在大字号下的崩溃风险，需在 handoff 前用 WCAG AA 体检一遍。

4. **`/design:design-handoff`（最后，且仅当 IA 已冻结）**
   理由：handoff 产出 token/组件 props/断点/边界/动效规格，但**前提是设计已收敛**。当前 IA 还分叉，过早 handoff 等于把矛盾固化进开发，**暂不建议**。

一句话路线：**先做产品决策 → `design-system` 收敛组件 → `ux-copy` 补状态文案 → `accessibility-review` 体检 → 冻结后 `design-handoff`。**
