## 二、本轮范围（Brief）

### A. 建立配对映射（先做，别跳）
- 两个文件夹**命名/数量不同**（设计 45 / 真机 ~30），按**屏的身份**配对，产出一张映射表：`设计文件 ↔ 真机文件 ↔ 屏名 ↔ 状态`。示例对：
  - `screenshots/00_采集_采集_Tab_-_主界面.png` ↔ `screenshots_device/00_采集主界面.png`
  - `screenshots/15_运行A_…就绪.png` ↔ `screenshots_device/05_…？` / `screenshots/16_运行B…采集中` ↔ `…运行页` …（按内容判断，不要硬按序号）
- 标注三类：**仅设计有**（未实现 / 真机难复现的态，如各种空态/错误态）、**仅真机有**（设计稿没有的新增，如 `29_深色…`、`30_设置_外观主题`）、**两者都有**（可逐屏对照）。

### B. 逐屏视觉对照（用视觉，看图比对）
- 对每对图，比对：**布局 / 间距 / 字阶字重 / 颜色与 token / 图标形状 / 组件状态 / 文案**。
- **注意**：真机图是 1080×2400 + MIUI 状态栏，设计图是 470 宽手机框 + 模拟状态栏——**比"语义/结构/组件"差异，别抠像素级与状态栏样式**。
- 产出**差异清单**，每条含：`屏 · 差异点 · 归类 · 严重度(高/中/低)`。

### C. 分类处置（核心）
- **① 直接修**：实现侧 bug / 缺失 / 明显偏离设计，且**收敛方向与 SPEC + 原型一致** → 改 app 收敛。
- **② 先确认（不自行改，`AskUserQuestion` 列给用户裁决）**——凡差异属于下列任一：
  1. **设计稿 ↔ SPEC 冲突**（两个真源打架）；
  2. **设计稿 ↔ 用户先前已定口径冲突**（见 §D「别乱改清单」，那是已确认的决策，设计稿可能更旧）；
  3. **OS 约束导致无法照搬设计**（如深色模式 splash、系统栏）；
  4. 改动会**牵动已定行为 / 信息架构 / 交互**（不是纯视觉微调）。
- 每个待确认项给：`屏 · 差异 · 设计稿这边 vs 现状这边 · 推荐选项 + 理由`，让用户一句话能定。

### D. 「别乱改」清单（已确认/有意为之；除非用户当面改口，不要按设计稿"修回去"）
- **启动屏两层落地**（§5.1）：系统 `SplashScreen` 只画底色、**图标设透明**（避免与应用内 logo 形成"两个开屏"）；**深色模式品牌深底 `#11151C`**（`values-night`，因 force-dark 拦不住）；logo+字标+副标+三点由 **应用内 `AppSplash`** 出。→ 真机 splash 与浅色设计稿"不同"是**对的**。
- **全局外观模式 亮/暗/跟随系统**（§8.4，新增，原型只有浅色）：**深色界面 ≠ 浅色设计稿，不要拿浅色稿去"修"深色屏**；只查深色自身的对比度/截断。
- **V4 信息架构与交互**：底部三 Tab；**扁平设备列表 + 限额**（DUT≤3 + 参考≤1，**不是** `Map<Role>`）；采集运行**硬锁定** + 长按 2 秒结束；在场感 = **前台服务通知**（无 app 内 Banner）；实时流 `[unixMs] HEX`；标签 Pin + Start/Stop；暂停 = 仅停滚动。
- **暂不实现 / 占位**：配置A/B（传感器总控 / 设备端算法）；设备维护(DUT) 四子项；服务器同步/远程下发（二期·灰显）。这些"空/灰"是对的，别去补功能。
- 一句话：**这些地方真机与设计稿不一致，是已决策的结果，不要"自动收敛"——要动先问。**

### E. 修完即验证（真机优先）
- 每修一块 → `./gradlew :app:assembleDebug` → 真机**重抓该屏** → 再对照。
- **截图配方（已验证可用）**：
  - adb：`C:\Users\260164\AppData\Local\Android\Sdk\platform-tools\adb.exe`，设备 `-s 5d510bb4`，包名 `io.bluetrace`，主 Activity `io.bluetrace/.MainActivity`。
  - 普通屏：`am force-stop` → `am start` →（等过启动屏）→ `screencap -p /sdcard/x.png` → `adb pull`（**别用 `exec-out > file`，PowerShell 会损坏二进制**）。
  - 导航靠 `input tap X Y`（坐标系 1080×2400，从截图估算）/ `input swipe` / `input keyevent KEYCODE_BACK`；长按结束用 `input swipe X Y X Y 2200`。Mock BLE 可连设备跑完整流程到运行/结束/数据。
  - **深/浅色**：`adb shell cmd uimode night yes|no`（测"跟随系统"）；外观模式也可走 设置→外观 切。
  - **启动屏**：只在冷启动一闪——`force-stop` 制造冷启动 + `am start --activity-no-animation` + 无 sleep 高速连拍 ~30 帧再挑（普通 launch 截不到）。
  - 看图判断时把全尺寸图**缩略到 ~400 宽**再 Read，避免一次性多张全尺寸触发图片 API 限制。
- 改完把对应真机图**更新**到 `screenshots_device/`（覆盖旧的），保持对照集最新。

## 三、不做
- 不动通信协议（继续 Mock）；不重构无关模块；不擅自改 V4 已定 IA/交互/口径（要动先问，§C②）。

- 不追求像素级 1:1 与状态栏样式一致（比语义/结构）。

## 四、验收清单（Definition of Done · `/goal` 达成条件）
1. **配对映射表** + **全屏差异清单**（含归类/严重度）产出（可写入 `Docs/` 一个临时对照笔记或提交说明）。
2. **「直接修」类全部收敛**：真机重抓与设计稿一致（或在 SPEC 允许的实现差范围内）。
3. **「先确认」类全部以 `AskUserQuestion` 提交用户**、按裁决落地，**无擅自更改**已定口径（§D）。
4. **深色模式各屏自查**：无对比度不足 / 文本截断 / 图标不可见；外观三档（跟随系统/浅色/深色）切换即时生效不回归。
5. **构建通过**（`:app:assembleDebug` + `:shared` JVM 单测全绿）；真机闭环（扫描→连接→采集→结束→数据→导出）不回归。
6. **截图集更新**：被修正的屏在 `screenshots_device/` 重拍覆盖；偏离 SPEC 处在提交说明里列明。

> 偏离 SPEC/原型处在提交说明里列出并说明原因。本轮聚焦"视觉/交互对照收敛"，不引入新功能（外观模式等已在 v2/本前置工作完成）。
