BlueTrace 架构评估：现状、问题与演进（2026-07-06）

输入：四路代码审查（代码审查报告，波次①–④已修复高/中危主体）+ 本轮结构层审计（模块/包依赖矩阵、KMP 边界、DI、协程纪律逐项核对，file:line 佐证）。 定位：回答"当前架构有什么问题、离最佳实践差在哪、M7/M8 来临前该怎么演进"。方案衔接 02 协议注册架构设计，不另起炉灶。

1. 一句话结论

骨架是对的，值得保持；问题集中在四处：一个真正的线程模型错误（采集落盘跑在 CPU 池）、两个包级依赖环（Mock 混放接口包）、一处状态双真相（ConnectionRegistry vs linkState）、一批"该在 shared 却写在 app"的 iOS 债。 M7 的协议架构不必新设计——02 号注册式设计成立，且 R1–R3 不等协议冻结即可落码。

4. 问题清单（按优先级）

P0 · 线程与正确性（本周就修，半天）

# 问题 证据 改法
A1 采集落盘跑在 Dispatchers.Default：会话 runScope 继承应用级 scope（Default CPU 池），okio 阻塞写盘在计算线程上执行；真实 BLE 数百包/秒时挤占 CPU 池、写延迟放大 AppModule.kt:53 + DefaultSessionController.kt:125-140 会话 scope 注入独立 IO 派发器（named("sessionIo")，Dispatchers.IO.limitedParallelism(1) 保持单线程语义不变）
A2 应用级 scope 与 logWriter scope 均无 CoroutineExceptionHandler——未捕获异常直接崩进程（波次①只兜了消费循环这一处） 全仓 0 处 handler（审计 §5） 两个 scope 补 handler → DiagnosticsLog 记 ERROR
A3 IO 边界不统一：SessionStore/DeviceLogStore 是阻塞 API，withContext(IO) 散落 7 个调用点、3 处直接写在 Composable 里；采集主页 start 路径仍在主线程建目录写 manifest（审查架构#4） 审计 §6；ConsoleLogListScreen.kt:46 等 两个 Store 的公开方法改 suspend + 内部 withContext(注入 io)，调用点删光散落切换

P1 · 结构边界（M7 开工前修，约一天）

# 问题 证据 改法
B1 包级双向环 ×2：ble↔protocol、ble↔s7，成因都是 Mock 实现与接口同包混放（MockBleClient 还内嵌 S7MockWatch） MockBleClient.kt:9-10 vs SampleDecoder.kt:3、S7Console.kt:3 Mock 迁 ble.mock 子包：mock→{ble,protocol,s7} 单向，环消失；顺带为 02 的 R2（MockBleProfile 化）铺路
B2 已连接状态双真相：ConnectionRegistry 与 BleClient.linkState 靠各 VM 手动对齐，漏一处就出幽灵条目 ConsoleConnectViewModel.kt:137-142 等（审计 §7） Registry 下沉 shared 并改为事件驱动：订阅 linkState，DISCONNECTED 自动清退；VM 只调 connect/disconnect
B3 iOS 债（纯 Kotlin 写在 app）：ConnectionRegistry、CollectDraft、Subject→S7Person 映射、refreshAll 聚合编排 审计 §4/§8 全部下沉 commonMain（Registry 随 B2 一起）；Exporter 的 zip 组包逻辑拆纯函数下沉，MediaStore 落盘留 app
B4 协议接入仍是全局单一解码器，第三种协议进来即塌（DeviceKind 二分） AppModule.kt:84；02 §1 落地 02 设计 R1–R3（registry 骨架 + Mock Profile 化 + ProtocolEvent），纯 shared 重构、单测护航、不等协议冻结

P2 · 工程卫生（顺手/择机）

# 问题 改法

5. 演进路线（与 02 设计对齐）

建议执行顺序：P0（A1–A3，半天）→ P1（B1–B4，约一天，其中 B4=02 的 R1–R3）→ R4 真实链路首连（等心率带）→ M7 解码器（等冻结）。P2 里 C6（CI）建议立刻配，其余随手。

6. 需要拍板的三个决策

# 决策 选项与建议
D1 02 设计 R1–R3 是否现在落码 建议：是。纯 shared 重构、测试护航、消除"第三种协议即塌"的结构风险，且是 M7 的必经路
D2 采集档 BLE 传输：Nordic Kotlin-BLE vs 自研 AndroidBleClient 补五件事（写队列/自动重连/notify 背压/GATT 串行化/MTU 分包） 倾向 Nordic（五件事都是它的本职，自研补齐≈重写一个库）；控制台档现状够用不动
D3 CI 是否上 建议：是（GitHub Actions，push 跑 jvmTest+assembleDebug，防"只在我机器上绿"）
